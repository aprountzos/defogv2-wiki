Installing Linkerd

Overview​

In this guide, you'll install Linkerd on all three clusters. Each cluster will get:

What you'll install:

  • Linkerd CRDs - Custom resource definitions
  • Control Plane - Core Linkerd components
  • Linkerd Viz - Dashboard and metrics
  • Multicluster - Cross-cluster communication

Prerequisites​

Before starting:

  • ✅ Three K3d clusters running (top, mid, bot)
  • ✅ kubectl configured with access to all clusters
  • ✅ Linkerd CLI installed
  • ✅ step CLI installed (for certificates)

Install Linkerd CLI​

If you haven't installed the Linkerd CLI yet:

# Install specific version
export LINKERD2_VERSION=stable-2.14.10
curl --proto '=https' --tlsv1.2 -sSfL https://run.linkerd.io/install | sh

# Add to PATH
export PATH=$PATH:$HOME/.linkerd2/bin

# Verify installation
linkerd version

Expected output:

Client version: stable-2.14.10
Server version: unavailable

Install step CLI​

For certificate generation:

  • macOS
  • Linux
  • Windows
brew install step
wget https://dl.step.sm/gh-release/cli/docs-ca-install/v0.25.0/step-cli_0.25.0_amd64.deb
sudo dpkg -i step-cli_0.25.0_amd64.deb

Download from GitHub Releases

Verify:

step version

Understanding the Installation​

Why Certificates?​

Linkerd uses mTLS (mutual TLS) for all service-to-service communication. This requires:

  1. Trust Anchor (Root CA) - Shared across all clusters
  2. Identity Issuer (Intermediate CA) - Unique per cluster

Why this matters:

  • All clusters trust the same root
  • Each cluster issues its own certificates
  • Proxies can verify each other across clusters

Installation Flow​

For each cluster, we'll:

Step 1: Generate Trust Anchor​

Create a shared root certificate for all clusters.

# Create temporary directory for certificates
mkdir -p ~/linkerd-certs
cd ~/linkerd-certs

# Generate root certificate (trust anchor)
step certificate create \
  root.linkerd.cluster.local \
  ca.crt ca.key \
  --profile root-ca \
  --no-password \
  --insecure \
  --force

What this creates:

  • ca.crt - Root certificate (public)
  • ca.key - Root private key (keep secret!)

Verify:

step certificate inspect ca.crt

You should see:

Certificate:
    Subject:     root.linkerd.cluster.local
    Issuer:      root.linkerd.cluster.local
    ...

Step 2: Install Linkerd on Top Cluster​

Step 2.1: Generate Cluster Certificate​

Create an identity issuer certificate for the top cluster:

step certificate create \
  identity.linkerd.top.k3d.example.com \
  top-issuer.crt top-issuer.key \
  --ca ca.crt \
  --ca-key ca.key \
  --profile intermediate-ca \
  --not-after 8760h \
  --no-password \
  --insecure

Step 2.2: Pre-flight Check​

Verify the cluster is ready for Linkerd:

linkerd check --pre --context k3d-top

All checks should pass with ✔:

kubernetes-api
--------------
✔ can initialize the client
✔ can query the Kubernetes API

kubernetes-version
------------------
✔ is running the minimum Kubernetes API version
...

If any checks fail, resolve them before continuing.

Step 2.3: Install CRDs​

linkerd install --crds --context k3d-top | \
  kubectl apply --context k3d-top -f -

Expected output:

customresourcedefinition.apiextensions.k8s.io/servers.policy.linkerd.io created
customresourcedefinition.apiextensions.k8s.io/serviceprofiles.linkerd.io created
...

Step 2.4: Install Control Plane​

linkerd install \
  --context k3d-top \
  --cluster-domain top.k3d.example.com \
  --identity-trust-domain top.k3d.example.com \
  --identity-trust-anchors-file ca.crt \
  --identity-issuer-certificate-file top-issuer.crt \
  --identity-issuer-key-file top-issuer.key | \
  kubectl apply --context k3d-top -f -

What each flag does:

FlagPurpose
--cluster-domainDNS domain for this cluster
--identity-trust-domainTrust domain for mTLS
--identity-trust-anchors-fileRoot CA certificate
--identity-issuer-certificate-fileCluster's issuer cert
--identity-issuer-key-fileCluster's issuer key

Step 2.5: Wait for Linkerd to be Ready​

# Watch pods starting
kubectl get pods -n linkerd --context k3d-top --watch

# Wait for all pods to be ready (Ctrl+C when done)

Step 2.6: Verify Installation​

linkerd check --context k3d-top

All checks should pass! You should see:

linkerd-existence
-----------------
✔ 'linkerd-config' config map exists
✔ heartbeat ServiceAccount exist
✔ control plane replica sets are ready
...

Status check results are ✔

Step 2.7: Install Viz Extension​

linkerd viz install --context k3d-top | \
  kubectl apply --context k3d-top -f -

Wait for Viz to be ready:

kubectl wait --for=condition=ready pod \
  --all -n linkerd-viz \
  --context k3d-top \
  --timeout=300s

Verify:

linkerd viz check --context k3d-top

Step 2.8: Install Multicluster Extension​

linkerd multicluster install --context k3d-top | \
  kubectl apply --context k3d-top -f -

Wait for gateway to be ready:

kubectl wait --for=condition=ready pod \
  -n linkerd-multicluster \
  -l app=linkerd-gateway \
  --context k3d-top \
  --timeout=120s

Verify:

linkerd multicluster check --context k3d-top

Step 3: Install MetalLB on Top Cluster​

Linkerd multicluster gateways need LoadBalancer IPs. We'll use MetalLB.

Step 3.1: Install MetalLB​

kubectl apply --context k3d-top -f \
  https://raw.githubusercontent.com/metallb/metallb/v0.14.8/config/manifests/metallb-native.yaml

Step 3.2: Wait for MetalLB​

kubectl wait --namespace metallb-system \
  --for=condition=ready pod \
  --selector=app=metallb \
  --context k3d-top \
  --timeout=120s

Step 3.3: Configure IP Pool​

First, find your Docker network subnet:

docker network inspect multicluster-example \
  --format '{{range .IPAM.Config}}{{.Subnet}}{{end}}'

Example output: 172.18.0.0/16

Get the top cluster's server IP:

docker inspect k3d-top-server-0 \
  --format '{{range .NetworkSettings.Networks}}{{.IPAddress}}{{end}}'

Example output: 172.18.0.3

Create IP pool (adjust IPs based on your subnet):

kubectl apply --context k3d-top -f - <<EOF
apiVersion: metallb.io/v1beta1
kind: IPAddressPool
metadata:
  name: default-pool
  namespace: metallb-system
spec:
  addresses:
  - 172.18.0.200-172.18.0.210
---
apiVersion: metallb.io/v1beta1
kind: L2Advertisement
metadata:
  name: default
  namespace: metallb-system
spec:
  ipAddressPools:
  - default-pool
EOF
IP Range

Choose an IP range within your Docker network subnet that won't conflict with existing containers. The .200-.210 range is usually safe.

Step 3.4: Verify Gateway Has IP​

kubectl get svc -n linkerd-multicluster linkerd-gateway --context k3d-top

You should see an EXTERNAL-IP assigned:

NAME              TYPE           CLUSTER-IP      EXTERNAL-IP    PORT(S)
linkerd-gateway   LoadBalancer   10.43.xxx.xxx   172.18.0.200   4143:xxxxx/TCP

Step 6: Verify All Installations​

Check All Clusters​

Run checks on all clusters:

  • Top Cluster
  • Mid Cluster
  • Bot Cluster
linkerd check --context k3d-top
linkerd viz check --context k3d-top
linkerd multicluster check --context k3d-top
linkerd check --context k3d-mid
linkerd viz check --context k3d-mid

linkerd viz check --context k3d-bot

All checks should pass!

View Gateway IPs​

echo "=== Gateway IPs ==="
echo -n "Top: "
kubectl get svc -n linkerd-multicluster linkerd-gateway --context k3d-top -o jsonpath='{.status.loadBalancer.ingress[0].ip}'

echo -n "Mid: "
kubectl get svc -n linkerd-multicluster linkerd-gateway --context k3d-mid -o jsonpath='{.status.loadBalancer.ingress[0].ip}'

kubectl get svc -n linkerd-multicluster linkerd-gateway --context k3d-bot -o jsonpath='{.status.loadBalancer.ingress[0].ip}'

Understanding What You Installed​

Linkerd Components per Cluster​

Resource Usage​

Check how much each component uses:

kubectl top pods -n linkerd --context k3d-top
kubectl top pods -n linkerd-viz --context k3d-top
kubectl top pods -n linkerd-multicluster --context k3d-top

Typical usage:

  • Control plane: ~200-300MB RAM total
  • Viz: ~500-700MB RAM total
  • Multicluster: ~50-100MB RAM total

Troubleshooting​

Pre-flight Checks Fail​

If linkerd check --pre fails:

Missing CRDs:

# Check if CRDs exist
kubectl get crds | grep linkerd

Kubernetes version too old:

kubectl version --short

K3s should be 1.21+.

Control Plane Won't Start​

Check pod status:

kubectl get pods -n linkerd --context k3d-top

View pod logs:

kubectl logs -n linkerd -l linkerd.io/control-plane-component=identity --context k3d-top

Common issues:

  • Certificate problems: Regenerate certificates
  • Resource limits: Check kubectl top nodes
  • Image pull failures: Check internet connection

Gateway Has No IP​

Check MetalLB pods:

kubectl get pods -n metallb-system --context k3d-top

View MetalLB logs:

kubectl logs -n metallb-system -l app=metallb --context k3d-top

Check IP pool:

kubectl get ipaddresspool -n metallb-system --context k3d-top

Recreate if needed:

kubectl delete ipaddresspool default-pool -n metallb-system --context k3d-top
# Then recreate with correct IPs

Viz Dashboard Won't Load​

Check Viz pods:

kubectl get pods -n linkerd-viz --context k3d-top

Try accessing directly:

linkerd viz dashboard --context k3d-top

For each cluster:

# 1. Generate cluster certificate
step certificate create identity.linkerd.CLUSTER.k3d.example.com \

linkerd check --context k3d-CLUSTER

linkerd viz install --context k3d-CLUSTER | kubectl apply --context k3d-CLUSTER -f -
