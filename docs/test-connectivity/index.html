Testing Connectivity

Overview​

Now that your clusters are linked, let's deploy applications and verify everything works correctly.

Testing strategy:

  1. Deploy services in each cluster
  2. Test local communication
  3. Test cross-cluster communication
  4. Test traffic splitting
  5. View metrics and topology

Prerequisites​

  • ✅ All three clusters running with Linkerd
  • ✅ Clusters linked (Bot→Mid, Mid→Top)
  • ✅ All linkerd multicluster check passing

Test 1: Basic Service Deployment​

Deploy Hello Services in All Clusters​

We'll deploy a simple echo service in each cluster to test connectivity.

  • Top Cluster
  • Mid Cluster
  • Bot Cluster
# Create and annotate namespace
kubectl create namespace demo --context k3d-top
kubectl annotate namespace demo linkerd.io/inject=enabled --context k3d-top

# Deploy hello service
kubectl apply --context k3d-top -n demo -f - <<EOF
apiVersion: apps/v1
kind: Deployment
metadata:
  name: hello
spec:
  replicas: 2
  selector:
    matchLabels:
      app: hello
  template:
    metadata:
      labels:
        app: hello
        version: top
    spec:
      containers:
      - name: hello
        image: hashicorp/http-echo
        args:
        - "-text=Hello from TOP cluster"
        - "-listen=:8080"
        ports:
        - containerPort: 8080
---
apiVersion: v1
kind: Service
metadata:
  name: hello
  labels:
    app: hello
    to-export: "true"
spec:
  selector:
    app: hello
  ports:
  - name: http
    port: 8080
    targetPort: 8080
EOF
# Create and annotate namespace
kubectl create namespace demo --context k3d-mid
kubectl annotate namespace demo linkerd.io/inject=enabled --context k3d-mid

# Deploy hello service
kubectl apply --context k3d-mid -n demo -f - <<EOF
apiVersion: apps/v1
kind: Deployment
metadata:
  name: hello
spec:
  replicas: 2
  selector:
    matchLabels:
      app: hello

    metadata:
      labels:
        app: hello
        version: mid

      containers:
      - name: hello

        args:
        - "-text=Hello from MID cluster"
        - "-listen=:8080"

        - containerPort: 8080
---

    app: hello
    to-export: "true"
spec:
  selector:
    app: hello
  ports:
  - name: http
    port: 8080
    targetPort: 8080
EOF

kubectl create namespace demo --context k3d-bot

Wait for Deployments​

# Top
kubectl wait --for=condition=ready pod \
  -l app=hello -n demo --context k3d-top --timeout=60s

# Mid
kubectl wait --for=condition=ready pod \
  -l app=hello -n demo --context k3d-mid --timeout=60s

# Bot
kubectl wait --for=condition=ready pod \
  -l app=hello -n demo --context k3d-bot --timeout=60s

Verify Linkerd Injection​

Check that proxies were injected:

# Should show 2/2 (app + proxy)
kubectl get pods -n demo --context k3d-top
kubectl get pods -n demo --context k3d-mid
kubectl get pods -n demo --context k3d-bot

Expected output:

NAME                     READY   STATUS    RESTARTS   AGE
hello-xxxxxxxxxx-xxxxx   2/2     Running   0          30s
hello-xxxxxxxxxx-xxxxx   2/2     Running   0          30s

Test 2: Local Communication​

Test that services work within their own cluster first.

Deploy Test Pod in Bot​

kubectl apply --context k3d-bot -n demo -f - <<EOF
apiVersion: v1
kind: Pod
metadata:
  name: curl
spec:
  containers:
  - name: curl
    image: curlimages/curl
    command: ["sleep", "infinity"]
EOF

# Wait for pod
kubectl wait --for=condition=ready pod/curl \
  -n demo --context k3d-bot --timeout=60s

Test Local Service​

kubectl exec -it curl -n demo --context k3d-bot -- \
  curl -s http://hello:8080

Expected output:

Hello from BOT cluster

✅ Local communication works!

Test 3: Cross-Cluster Communication​

Check Mirrored Services​

# In Bot, check for mirrored services
kubectl get svc -n demo --context k3d-bot

Expected output:

NAME        TYPE        CLUSTER-IP      EXTERNAL-IP   PORT(S)    AGE
hello       ClusterIP   10.43.xxx.xxx   <none>        8080/TCP   2m
hello-mid   ClusterIP   10.43.xxx.xxx   <none>        8080/TCP   1m
hello-top   ClusterIP   10.43.xxx.xxx   <none>        8080/TCP   1m

You should see:

  • hello - Local service
  • hello-mid - Mirrored from Mid
  • hello-top - Mirrored from Top (via Mid)

Test Bot → Mid Communication​

kubectl exec -it curl -n demo --context k3d-bot -- \
  curl -s http://hello-mid:8080

Expected:

Hello from MID cluster

✅ Bot can reach Mid!

Test Bot → Top Communication​

kubectl exec -it curl -n demo --context k3d-bot -- \
  curl -s http://hello-top:8080

Expected:

Hello from TOP cluster

✅ Bot can reach Top (through Mid)!

Test All Services​

Run multiple requests to see all clusters respond:

echo "Testing local service:"
kubectl exec -it curl -n demo --context k3d-bot -- curl -s http://hello:8080

echo ""
echo "Testing Mid service:"
kubectl exec -it curl -n demo --context k3d-bot -- curl -s http://hello-mid:8080

echo ""
echo "Testing Top service:"
kubectl exec -it curl -n demo --context k3d-bot -- curl -s http://hello-top:8080

Output:

Testing local service:
Hello from BOT cluster

Testing Mid service:
Hello from MID cluster

Testing Top service:
Hello from TOP cluster

Perfect! All three clusters are communicating! 🎉

Test 4: Verify Secure Communication​

Check mTLS is Active​

# Check that traffic is encrypted
linkerd viz tap deployment/hello -n demo --context k3d-bot

You should see tls=true for all connections.

View Service Mesh Topology​

# Open Linkerd dashboard
linkerd viz dashboard --context k3d-bot

In the dashboard:

  1. Navigate to Namespaces → demo
  2. Click on the hello deployment
  3. View Inbound and Outbound metrics
  4. Check that Secured shows 100%

Test 5: Traffic Splitting​

Let's test traffic splitting between local and remote services.

Create TrafficSplit​

kubectl apply --context k3d-bot -n demo -f - <<EOF
apiVersion: split.smi-spec.io/v1alpha1
kind: TrafficSplit
metadata:
  name: hello-split
spec:
  service: hello
  backends:
  - service: hello
    weight: 500      # 50% to local (Bot)
  - service: hello-mid
    weight: 300      # 30% to Mid
  - service: hello-top
    weight: 200      # 20% to Top
EOF

Test Traffic Distribution​

Run multiple requests and see traffic distributed:

for i in {1..20}; do
  kubectl exec curl -n demo --context k3d-bot -- curl -s http://hello:8080
  sleep 0.5
done

You should see responses from all three clusters in roughly 50%/30%/20% distribution.

View Traffic Split Metrics​

linkerd viz stat trafficsplit -n demo --context k3d-bot

Expected output:

NAME          APEX    LEAF          WEIGHT   SUCCESS   RPS
hello-split   hello   hello         500      100.00%   2.5rps
hello-split   hello   hello-mid     300      100.00%   1.5rps
hello-split   hello   hello-top     200      100.00%   1.0rps

Test 6: Load Testing​

Run Load Test​

Generate some load to see metrics in action:

# Generate 100 requests
kubectl exec curl -n demo --context k3d-bot -- sh -c '
  for i in $(seq 1 100); do
    curl -s http://hello:8080 > /dev/null
    echo -n "."
  done
  echo " Done!"
'

View Live Metrics​

# View deployment stats
linkerd viz stat deploy -n demo --context k3d-bot

# View traffic in real-time
linkerd viz tap deploy/hello -n demo --context k3d-bot

Check Success Rates​

linkerd viz stat deploy/hello -n demo --context k3d-bot

Expected output:

NAME    MESHED   SUCCESS   RPS   LATENCY_P50   LATENCY_P95   LATENCY_P99
hello   2/2      100.00%   5.0rps       1ms           2ms           3ms

Test 7: Service Profiles​

Create service profiles for better reliability.

Create Service Profile​

kubectl apply --context k3d-bot -n demo -f - <<EOF
apiVersion: linkerd.io/v1alpha2
kind: ServiceProfile
metadata:
  name: hello.demo.svc.cluster.local
spec:
  routes:
  - name: GET /
    condition:
      method: GET
      pathRegex: /
    timeout: 5s
    retries:
      limit: 3
      timeout: 1s
EOF

Test Retry Logic​

# This should automatically retry on failure
kubectl exec curl -n demo --context k3d-bot -- \
  curl -s http://hello:8080

Test 8: Cross-Namespace Communication​

Create Another Namespace​

# In Bot cluster
kubectl create namespace prod --context k3d-bot
kubectl annotate namespace prod linkerd.io/inject=enabled --context k3d-bot

# Deploy test pod
kubectl apply --context k3d-bot -n prod -f - <<EOF
apiVersion: v1
kind: Pod
metadata:
  name: curl
spec:
  containers:
  - name: curl
    image: curlimages/curl
    command: ["sleep", "infinity"]
EOF

kubectl wait --for=condition=ready pod/curl -n prod --context k3d-bot --timeout=60s

Test Cross-Namespace Call​

# Call service in demo namespace from prod namespace
kubectl exec curl -n prod --context k3d-bot -- \
  curl -s http://hello.demo:8080

Expected:

Hello from BOT cluster

✅ Cross-namespace communication works!

Test Mirrored Services Cross-Namespace​

# Call mirrored services
kubectl exec curl -n prod --context k3d-bot -- curl -s http://hello-mid.demo:8080
kubectl exec curl -n prod --context k3d-bot -- curl -s http://hello-top.demo:8080

Test 9: Observability​

View Service Topology​

linkerd viz dashboard --context k3d-bot

What to check:

  1. Namespaces - All namespaces meshed
  2. Deployments - All show 100% success rate
  3. Edges - See connections between services
  4. Live Calls - Real-time traffic

Check Prometheus Metrics​

# Forward Prometheus port
kubectl port-forward -n linkerd-viz svc/prometheus 9090:9090 --context k3d-bot &

# Open browser to localhost:9090

Query examples:

  • request_total - Total requests
  • response_latency_ms_bucket - Latency distribution
  • outbound_tcp_open_connections - Active connections

View Grafana Dashboards​

# Forward Grafana port
kubectl port-forward -n linkerd-viz svc/grafana 3000:3000 --context k3d-bot &

# Open browser to localhost:3000

Default dashboards:

  • Linkerd Top Line - Overall cluster metrics
  • Linkerd Deployment - Per-deployment metrics
  • Linkerd Pod - Per-pod metrics

Test 10: Failure Scenarios​

Test Service Failure​

Scale down Mid service to test failover:

# Scale to 0
kubectl scale deployment hello -n demo --replicas=0 --context k3d-mid

# Try calling from Bot
kubectl exec curl -n demo --context k3d-bot -- curl -s http://hello-mid:8080

Expected: Request fails (503 or timeout).

View in Dashboard​

linkerd viz stat deploy -n demo --context k3d-bot

You should see success rate drop for hello-mid.

Scale Back Up​

kubectl scale deployment hello -n demo --replicas=2 --context k3d-mid

# Wait for pods
kubectl wait --for=condition=ready pod -l app=hello -n demo --context k3d-mid --timeout=60s

# Test again
kubectl exec curl -n demo --context k3d-bot -- curl -s http://hello-mid:8080

Success rate should recover!

Test 11: Performance Testing​

Concurrent Requests​

kubectl exec curl -n demo --context k3d-bot -- sh -c '
  for i in $(seq 1 50); do
    curl -s http://hello:8080 &
  done
  wait
  echo "All requests completed"
'

Check Latency​

linkerd viz stat deploy/hello -n demo --context k3d-bot

Look at P50, P95, P99 latencies.

Check Gateway Performance​

linkerd viz stat deploy -n linkerd-multicluster --context k3d-bot

Gateway should show healthy metrics.

Troubleshooting Tests​

Request Fails with "Connection Refused"​

Check pod is running:

kubectl get pods -n demo --context k3d-bot

Check service exists:

kubectl get svc -n demo --context k3d-bot

Check endpoints:

kubectl get endpoints hello -n demo --context k3d-bot

Request Times Out​

Check Linkerd proxy:

kubectl logs <pod-name> -n demo -c linkerd-proxy --context k3d-bot --tail 20

Check gateway connectivity:

linkerd multicluster gateways --context k3d-bot

Test DNS:

kubectl exec curl -n demo --context k3d-bot -- nslookup hello-mid.demo.svc.cluster.local

Traffic Split Not Working​

Check TrafficSplit exists:

kubectl get trafficsplit -n demo --context k3d-bot

Verify apex service has no selector:

kubectl get svc hello -n demo --context k3d-bot -o yaml | grep selector

Should be empty or selector: {}.

Check SMI resources:

kubectl get crds | grep split

Remove Demo Namespace​

kubectl delete namespace demo --context k3d-top
