Executor

The Executor is the action layer of Defog v2. It executes all Kubernetes operations (deployments, traffic splits) and provides the API gateway for peer cluster communication.

Overview​

Purpose​

The Executor serves dual roles:

1. Local Operations (for Orchestrator)​

  • Create/delete Deployments
  • Scale workloads
  • Apply TrafficSplit CRDs
  • Validate operations
  • Provide audit logging

2. Peer API (for Remote Clusters)​

  • Check capacity
  • Handle deployment requests
  • Return cluster metrics
  • Coordinate cross-cluster operations

Architecture Components​

Core Components​

  1. Server - HTTP server handling both local and peer APIs
  2. DeploymentManager - Manages Kubernetes Deployment lifecycle
  3. TrafficSplitManager - Applies SMI TrafficSplit CRDs
  4. OperationValidator - Validates all incoming requests
  5. CapacityCalculator - Calculates cluster resource availability
  6. AuditLogger - Logs all operations for compliance

Dual API Design​

Deployment Management​

Creating Deployments​

Implementation Details​

The DeploymentManager handles the complete lifecycle:

type DeploymentManager struct {
    client    kubernetes.Interface
    validator *OperationValidator
    auditLog  *AuditLogger
}

Key Features:

  • Validates all requests before execution
  • Parses resource quantities (CPU/memory)
  • Creates Deployment with proper labels (app, managed-by, source-cluster)
  • Automatically creates corresponding Service
  • Logs all operations to audit log
  • Returns service endpoint for DNS resolution

Resource Management:

  • Supports CPU/memory requests and limits
  • Validates resource format (e.g., "500m", "512Mi")
  • Enforces foreign workload limits via validator

Labels Applied:

app: <service-name>
managed-by: defog-v2
source-cluster: <cluster-id>  # For peer workloads

TrafficSplit Management​

Applying Traffic Splits​

The TrafficSplitManager uses Kubernetes dynamic client to manage SMI TrafficSplit CRDs:

type TrafficSplitManager struct {
    dynamicClient dynamic.Interface
    auditLog      *AuditLogger
}

Process:

  1. Validates backend weights sum to 100
  2. Builds unstructured TrafficSplit object
  3. Attempts update (if exists) or create (if new)
  4. Logs operation to audit

TrafficSplit Structure:

apiVersion: split.smi-spec.io/v1alpha1
kind: TrafficSplit
metadata:
  name: <service-name>
  namespace: <namespace>
spec:
  service: <service-name>
  backends:
    - service: api
      weight: 70
    - service: api-v2
      weight: 30

Validation​

Request Validator​

The OperationValidator ensures all requests meet requirements:

type OperationValidator struct {
    config *Config
}

Validation Checks:

  1. ✅ Service name required
  2. ✅ Namespace required
  3. ✅ Image required
  4. ✅ Replicas >= 1
  5. ✅ CPU request required and valid format
  6. ✅ Foreign workload CPU limit enforcement

Foreign Workload Protection:

  • Checks if SourceCluster != ClusterID
  • Enforces MaxForeignCPU limit
  • Prevents peer clusters from overwhelming local cluster

Capacity Management​

Capacity Calculator​

The CapacityCalculator queries Kubernetes nodes for resource availability:

type CapacityCalculator struct {
    client kubernetes.Interface
    config *Config
}

Metrics Calculated:

  • Total CPU across all nodes (in cores)
  • Total Memory across all nodes (in GiB)
  • Free/Allocatable CPU
  • Free/Allocatable Memory
  • Schedulable pods (estimated)

Usage:

  • Peer API uses this for capacity checks
  • Determines if deployment requests can be satisfied
  • Supports LFU eviction decisions (orchestrator integration)

Peer API​

Handling Peer Requests​

Can-Deploy Logic​

The executor determines if it can accept a peer workload:

  1. Already deployed? → Return already_have_it
  2. Check capacity → Query nodes for free resources
  3. Sufficient resources? → Return will_deploy
  4. Insufficient resources? → Return cannot_deploy

Note: LFU eviction logic is handled by the orchestrator, not the executor. The executor only reports capacity.

Security​

Authentication​

All Peer API endpoints require Bearer token authentication:

func (s *Server) authenticate(next http.HandlerFunc) http.HandlerFunc {
    // Validates Authorization header
    // Checks against configured AUTH_TOKEN
    // Returns 401 if invalid
}

Protected Endpoints:

  • /peer/v1/capacity
  • /peer/v1/can-deploy
  • /peer/v1/deploy
  • /peer/v1/services/\{name\}

Public Endpoints:

  • /peer/v1/health (for monitoring)

Configuration​

Security settings via environment variables:

AUTH_TOKEN=your-secure-token-here
CLUSTER_ID=cluster-a
MAX_FOREIGN_CPU=2000m
PORT=8080

API Reference​

Local API​

POST /api/v1/deployments​

Create a deployment.

Request:

{
  "serviceName": "api",
  "namespace": "default",
  "image": "myapp:v1.0.0",
  "replicas": 2,
  "resources": {
    "cpuRequest": "500m",
    "memoryRequest": "512Mi",
    "cpuLimit": "1000m",
    "memoryLimit": "1Gi"
  },
  "sourceCluster": "cluster-b"
}

Response:

{
  "success": true,
  "serviceName": "api",
  "endpoint": "api.default.svc.cluster.local",
  "message": "Deployment created successfully"
}

DELETE /api/v1/deployments/{name}​

Delete a deployment and its service.

Response:

{
  "status": "deleted",
  "message": "Service api deleted"
}

PATCH /api/v1/deployments/{name}/scale​

Scale a deployment.

Request:

{
  "replicas": 5
}

Response:

{
  "status": "scaled",
  "replicas": 5
}

POST /api/v1/traffic-splits​

Apply traffic split.

Request:

{
  "serviceName": "api",
  "namespace": "default",
  "backends": [
    {"service": "api", "weight": 70},
    {"service": "api-v2", "weight": 30}
  ]
}

Response:

{
  "success": true,
  "message": "Traffic split applied successfully"
}

Validation:

  • Backend weights must sum to exactly 100
  • Returns error if sum != 100

Peer API​

All peer endpoints require Authorization: Bearer <token> header.

GET /peer/v1/capacity​

Get cluster capacity metrics.

Response:

{
  "clusterId": "cluster-a",
  "freeCpu": 4.5,
  "freeMemory": 8.2,
  "totalCpu": 8.0,
  "totalMemory": 16.0,
  "schedulablePods": 110
}

POST /peer/v1/can-deploy​

Check if can deploy service.

Request:

{
  "serviceName": "payments",
  "namespace": "default",
  "resources": {
    "cpuRequest": "500m",
    "memoryRequest": "512Mi"
  },
  "sourceCluster": "cluster-b",
  "currentRps": 450.0
}

Response:

{
  "status": "will_deploy",
  "message": "Deployment initiated",
  "estimatedTime": "30s"
}

Status Values:

  • already_have_it - Service already exists locally
  • will_deploy - Will deploy the service
  • cannot_deploy - Insufficient capacity (see reason)

POST /peer/v1/deploy​

Deploy a service from peer cluster.

Request: Same as local deployment API

Response: Same as local deployment API

Validation:

  • Subject to MaxForeignCPU limit
  • Validates all resources
  • Labels with source-cluster

DELETE /peer/v1/services/{name}​

Delete a service deployed by peer.

Response:

{
  "status": "deleted"
}

GET /peer/v1/health​

Health check endpoint (no auth required).

Response:

{
  "status": "healthy",
  "clusterId": "cluster-a",
  "version": "v2.0.0"
}

Deployment​

Kubernetes Manifests​

apiVersion: v1
kind: ConfigMap
metadata:
  name: defog-config
  namespace: defog-system
data:
  CLUSTER_ID: "cluster-a"
  MAX_FOREIGN_CPU: "2000m"
  PORT: "8080"
---
apiVersion: v1
kind: Secret
metadata:
  name: defog-secrets
  namespace: defog-system
type: Opaque
stringData:
  AUTH_TOKEN: "your-secure-token-here"
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: executor
  namespace: defog-system
spec:
  replicas: 1
  selector:
    matchLabels:
      app: executor
  template:
    metadata:
      labels:
        app: executor
    spec:
      serviceAccountName: defog-controller
      containers:
      - name: executor
        image: registry.io/defog-executor:v2.0.0
        ports:
        - containerPort: 8080
          name: http
        env:
        - name: AUTH_TOKEN
          valueFrom:
            secretKeyRef:
              name: defog-secrets
              key: AUTH_TOKEN
        envFrom:
        - configMapRef:
            name: defog-config
        resources:
          requests:
            cpu: 200m
            memory: 256Mi
          limits:
            cpu: 500m
            memory: 512Mi
        livenessProbe:
          httpGet:
            path: /peer/v1/health
            port: 8080
          initialDelaySeconds: 10
          periodSeconds: 30
        readinessProbe:
          httpGet:
            path: /peer/v1/health
            port: 8080
          initialDelaySeconds: 5
          periodSeconds: 10
---
apiVersion: v1
kind: Service
metadata:
  name: executor
  namespace: defog-system
spec:
  type: LoadBalancer  # For peer access
  selector:
    app: executor
  ports:
  - port: 443
    targetPort: 8080
    name: https

RBAC Requirements​

apiVersion: v1
kind: ServiceAccount
metadata:
  name: defog-controller
  namespace: defog-system
---
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: executor-role
rules:
# Deployment management
- apiGroups: ["apps"]
  resources: ["deployments"]
  verbs: ["create", "delete", "patch", "get", "list", "update"]

# Service management
- apiGroups: [""]
  resources: ["services"]
  verbs: ["create", "delete", "get", "list"]

# TrafficSplit CRD
- apiGroups: ["split.smi-spec.io"]
  resources: ["trafficsplits"]
  verbs: ["create", "patch", "get", "list", "update"]

# Capacity calculation
- apiGroups: [""]
  resources: ["nodes"]
  verbs: ["get", "list"]

- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get", "list"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: executor-binding
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: ClusterRole
  name: executor-role
subjects:
- kind: ServiceAccount
  name: defog-controller
  namespace: defog-system

Monitoring & Observability​

Logging​

All operations are logged with structured output:

[AUDIT] 2024-12-08T10:30:45Z - Operation: deploy - Data: {serviceName:api namespace:default...}
[AUDIT] 2024-12-08T10:31:12Z - Operation: scale - Data: {name:api replicas:5}
[AUDIT] 2024-12-08T10:32:03Z - Operation: traffic-split - Data: {serviceName:api backends:[...]}

Metrics (Planned)​

The implementation is ready for Prometheus instrumentation:

Error Handling​

HTTP Error Codes​

  • 400 Bad Request - Invalid JSON or missing required fields
  • 401 Unauthorized - Missing or invalid auth token (peer API only)
  • 405 Method Not Allowed - Wrong HTTP method
  • 500 Internal Server Error - Kubernetes API errors or internal failures

Validation Errors​

All validation errors return descriptive messages:

{
  "error": "validation failed: CPU request exceeds max foreign CPU (2000m)"
}

Kubernetes Errors​

Kubernetes client errors are wrapped and returned:

{
  "error": "failed to create deployment: deployments.apps \"api\" already exists"
}

Best Practices​

  1. Always validate requests - Use the validator before any K8s operation
  2. Log all operations - Audit trail is critical for debugging
  3. Check capacity first - Verify resources before deploying
  4. Use structured labels - Enables proper service discovery and management
  5. Secure peer communication - Always use authentication tokens
  6. Monitor resource usage - Track executor's own resource consumption
  7. Handle errors gracefully - Return descriptive errors, don't panic

Implementation Notes​

What's Implemented​

✅ Full deployment lifecycle (create, delete, scale) ✅ Service creation alongside deployments ✅ TrafficSplit CRD management ✅ Request validation with foreign workload limits ✅ Capacity calculation from node metrics ✅ Peer API with authentication ✅ Audit logging for all operations ✅ Health check endpoint ✅ Proper error handling and HTTP status codes

What's Simplified​

⚠️ LFU Eviction Logic - The can-deploy endpoint checks capacity but doesn't implement eviction. That's the orchestrator's responsibility.

⚠️ Rollback Capability - Mentioned in docs but not fully implemented. Would require storing previous deployment state.

⚠️ Prometheus Metrics - Framework ready but metrics not instrumented yet.

⚠️ mTLS - Currently uses bearer token auth. Production should use mTLS for peer communication.

⚠️ Rate Limiting - Not implemented. Should be added to prevent peer API abuse.

Troubleshooting​

Common Issues​

Issue: failed to create k8s config: unable to load in-cluster configuration

  • Cause: Not running inside Kubernetes cluster
  • Solution: Use kubeconfig for local development or deploy to cluster

Issue: failed to create deployment: forbidden

  • Cause: ServiceAccount lacks RBAC permissions
  • Solution: Apply the ClusterRole and ClusterRoleBinding manifests

Issue: Unauthorized on peer API calls

  • Cause: Missing or incorrect AUTH_TOKEN
  • Solution: Ensure token matches on both clusters

Issue: validation failed: CPU request exceeds max foreign CPU

  • Cause: Peer workload exceeds MAX_FOREIGN_CPU limit
  • Solution: Increase limit or reduce peer request

Issue: Service endpoint not resolving

  • Cause: Service creation failed silently
  • Solution: Check logs for service creation errors
