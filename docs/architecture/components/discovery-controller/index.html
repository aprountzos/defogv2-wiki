Discovery Controller

The Discovery Controller is a Kubernetes operator that automatically ensures all services in specified namespaces are properly labeled for Linkerd multi-cluster export using event-driven architecture.

Overview​

Purpose​

The Discovery Controller serves as the entry point for service discovery in Defog v2. It ensures that:

  1. All services are exportable - Adds Linkerd export labels automatically
  2. Event-driven processing - Reacts to service creation/modification immediately
  3. Zero manual intervention - Services are auto-discovered on creation
  4. Audit trail - Events emitted when services are labeled

How It Works​

Event-Driven Architecture​

Labeling Logic​

The controller adds two labels to each service:

// Export label for Linkerd
svc.Labels["mirror.linkerd.io/exported"] = "true"

// Management label
svc.Labels["app.kubernetes.io/managed-by"] = "defog-v2"

Service Filtering​

Services are automatically skipped if they:

  • Start with "kubernetes" (e.g., kubernetes service)
  • Are in the kube-system namespace
  • Have annotation defog.io/skip-export: "true"
  • Are headless services (ClusterIP: None)

Configuration​

Command-Line Flags​

-namespaces         # Comma-separated namespaces (default: "default")
-export-label       # Label key (default: "mirror.linkerd.io/exported")
-export-value       # Label value (default: "true")
-managed-by         # Managed-by value (default: "defog-v2")
-orchestrator-url   # Orchestrator URL (default: "http://orchestrator:8080")
-http-port          # HTTP port (default: 8080)
-enable-events      # Enable event emission (default: true)
-v                  # Log verbosity (default: 0, max: 3)

ConfigMap Settings​

apiVersion: v1
kind: ConfigMap
metadata:
  name: discovery-controller-config
  namespace: defog-system
data:
  NAMESPACES: "default,production,staging"
  EXPORT_LABEL: "mirror.linkerd.io/exported"
  EXPORT_VALUE: "true"
  MANAGED_BY: "defog-v2"
  ORCHESTRATOR_URL: "http://orchestrator:8080"
  ENABLE_EVENTS: "true"
  HTTP_PORT: "8080"
  LOG_LEVEL: "2"

RBAC Requirements​

apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: discovery-controller-role
rules:
- apiGroups: [""]
  resources: ["services"]
  verbs: ["get", "list", "watch", "update", "patch"]
- apiGroups: [""]
  resources: ["namespaces"]
  verbs: ["get", "list"]

Events Emitted​

ServiceLabeledEvent​

Emitted when a service is successfully labeled:

{
  "type": "ServiceLabeled",
  "serviceName": "api",
  "namespace": "default",
  "action": "labeled",
  "labels": {
    "mirror.linkerd.io/exported": "true",
    "app.kubernetes.io/managed-by": "defog-v2"
  },
  "timestamp": "2024-01-15T10:30:00Z"
}

ServiceAlreadyLabeled​

Emitted when a service already has the correct labels:

{
  "type": "ServiceAlreadyLabeled",
  "serviceName": "api",
  "namespace": "default",
  "action": "already_labeled",
  "labels": {
    "mirror.linkerd.io/exported": "true"
  },
  "timestamp": "2024-01-15T10:30:00Z"
}

Deployment​

Kubernetes Manifest​

apiVersion: apps/v1
kind: Deployment
metadata:
  name: discovery-controller
  namespace: defog-system
spec:
  replicas: 1
  selector:
    matchLabels:
      app: discovery-controller
  template:
    metadata:
      labels:
        app: discovery-controller
    spec:
      serviceAccountName: discovery-controller
      containers:
      - name: discovery-controller
        image: your-registry.io/discovery-controller:v2.0.0
        args:
          - -namespaces=$(NAMESPACES)
          - -export-label=$(EXPORT_LABEL)
          - -export-value=$(EXPORT_VALUE)
          - -managed-by=$(MANAGED_BY)
          - -orchestrator-url=$(ORCHESTRATOR_URL)
          - -http-port=$(HTTP_PORT)
          - -enable-events=$(ENABLE_EVENTS)
          - -v=$(LOG_LEVEL)
        envFrom:
        - configMapRef:
            name: discovery-controller-config
        ports:
        - name: http
          containerPort: 8080
        resources:
          requests:
            cpu: 100m
            memory: 128Mi
          limits:
            cpu: 200m
            memory: 256Mi
        livenessProbe:
          httpGet:
            path: /health
            port: 8080
          initialDelaySeconds: 10
          periodSeconds: 30
        readinessProbe:
          httpGet:
            path: /ready
            port: 8080
          initialDelaySeconds: 5
          periodSeconds: 10
        securityContext:
          allowPrivilegeEscalation: false
          runAsNonRoot: true
          runAsUser: 65532
          capabilities:
            drop: ["ALL"]
          readOnlyRootFilesystem: true

Operational Characteristics​

Resource Usage​

MetricTypicalMaximum
CPU50m200m
Memory100Mi256Mi
API Watch Connections1 per namespaceN namespaces
Network< 1 Kbps< 5 Kbps

Performance​

  • Event processing: < 100ms per service
  • Initial sync: ~500ms per 100 services
  • Max services: 1000+ per namespace
  • Restart time: < 5 seconds
  • Watch reconnect: Automatic with exponential backoff

Troubleshooting​

Issue: Services Not Being Labeled​

Symptoms:

  • Services remain without export label
  • No log entries for the service

Diagnosis:

# Check controller is running
kubectl get pods -n defog-system -l app=discovery-controller

# Check logs with verbosity
kubectl logs -n defog-system deployment/discovery-controller

# Check if service is being watched
kubectl logs -n defog-system deployment/discovery-controller | grep "service-name"

# Verify RBAC permissions
kubectl auth can-i update services --as=system:serviceaccount:defog-system:discovery-controller

Common Causes:

Solutions:

  1. Controller not running:
kubectl rollout restart deployment/discovery-controller -n defog-system
  1. Namespace not watched:
# Update ConfigMap
data:
  NAMESPACES: "default,production,your-namespace"
  1. RBAC issues:
kubectl apply -f deployments/discovery-controller/rbac.yaml
  1. Service is filtered:
# Check if service matches skip conditions
kubectl get service <name> -o yaml | grep -E "ClusterIP|annotations"

Issue: Watch Connection Lost​

Symptoms:

  • Log shows "watch channel closed"
  • Services not being processed

Cause: The Kubernetes watch connection was interrupted

Solution: The controller automatically reconnects after 5 seconds. Check logs:

kubectl logs -n defog-system deployment/discovery-controller | grep "restarting"

Issue: Event Emission Failures​

Symptoms:

  • Services are labeled but orchestrator doesn't receive events
  • Logs show "Failed to send event to orchestrator"

Diagnosis:

# Check orchestrator connectivity
kubectl exec -n defog-system deployment/discovery-controller -- \
  wget -O- http://orchestrator:8080/health

# Check if events are disabled
kubectl get configmap discovery-controller-config -n defog-system -o yaml | grep ENABLE_EVENTS

Solutions:

  • Verify orchestrator service is running
  • Check orchestrator URL in ConfigMap
  • Event emission failures don't block service labeling (async operation)

Issue: High Memory Usage​

Symptoms:

  • Memory usage exceeds 256Mi
  • OOMKilled pod restarts

Causes:

  • Watching too many namespaces with many services
  • Large service objects

Solutions:

# Reduce watched namespaces
NAMESPACES: "production"  # Only critical namespace

# Or increase memory limits
resources:
  limits:
    memory: 512Mi

Advanced Configuration​

Multiple Namespace Watching​

The controller can watch multiple namespaces simultaneously:

data:
  NAMESPACES: "default,production,staging,dev"

Each namespace gets its own watch goroutine for isolation.

Custom Export Labels​

data:
  EXPORT_LABEL: "custom.io/exported"
  EXPORT_VALUE: "enabled"

Skipping Individual Services​

Add annotation to any service:

apiVersion: v1
kind: Service
metadata:
  name: internal-service
  annotations:
    defog.io/skip-export: "true"
spec:
  # ... service spec

Adjusting Log Verbosity​

data:
  LOG_LEVEL: "0"  # Errors only
  LOG_LEVEL: "1"  # + Warnings
  LOG_LEVEL: "2"  # + Info (recommended)
  LOG_LEVEL: "3"  # + Debug (verbose)

Best Practices​

1. Start Small​

Begin with a single namespace:

data:
  NAMESPACES: "development"

Then expand to production after validation.

2. Use Single Replica​

The controller is designed to run as a single replica (replicas: 1). Multiple replicas would:

  • Cause redundant labeling operations
  • Increase API server load unnecessarily
  • Provide no additional benefit (stateless operation)

3. Monitor Label Success​

Set up alerts for labeling errors:

rate(defog_discovery_errors_total[5m]) > 0

4. Enable Verbose Logging During Debugging​

# Temporarily increase verbosity
kubectl set env deployment/discovery-controller -n defog-system LOG_LEVEL=3

5. Use PodDisruptionBudget​

Prevent accidental downtime during cluster maintenance:

apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: discovery-controller
  namespace: defog-system
spec:
  maxUnavailable: 0
  selector:
    matchLabels:
      app: discovery-controller
