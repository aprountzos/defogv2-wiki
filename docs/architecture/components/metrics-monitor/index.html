Metrics Monitor

The Metrics Monitor continuously collects and analyzes metrics: pod stress (CPU/RAM usage), service RPS (requests per second), health metrics (error rates, latency), and cluster capacity. It uses adaptive thresholds and emits events when conditions are met, triggering the appropriate strategy.

Optimized for edge deployments - efficient resource usage while maintaining full observability.

Overview​

Purpose​

The Metrics Monitor is the observability layer of Defog v2, responsible for:

  1. Stress Monitoring - Detect when pods are overloaded (composite CPU+Memory score)
  2. Adaptive RPS Tracking - Measure requests per second with percentile-based thresholds
  3. Health Monitoring - Track error rates and latency per service
  4. Capacity Reporting - Calculate available cluster resources and identify hot nodes
  5. Event Emission - Notify Orchestrator when thresholds crossed
  6. History Tracking - Maintain time-series data for adaptive thresholds

Collection Strategy​

Concurrent Collection with Goroutines​

All metrics are collected in parallel for maximum efficiency:

Benefits:

  • 3x faster than sequential collection
  • Sub-second collection cycles
  • Efficient resource usage

Strategy 1: Stress Monitoring​

Tracks pod resource usage with composite scoring:

What it tracks:

  • CPU usage vs CPU limit (per pod)
  • Memory usage vs Memory limit (per pod)
  • Composite stress score (weighted: 70% CPU, 30% Memory)
  • Average stress across all pods

Key improvements:

  • ✅ Batch K8s API calls (all pods at once)
  • ✅ Informers/watches instead of polling
  • ✅ Configurable weights per service

Data structure:

type StressMetrics struct {
    Pods           []PodStress
    AvgCPUUsage    float64  // 0.0 - 1.0
    AvgMemoryUsage float64  // 0.0 - 1.0
    CompositeScore float64  // Weighted: CPU×0.7 + Mem×0.3
    StressLevel    string   // "low", "medium", "high", "critical"
}

Strategy 2: Adaptive RPS Monitoring​

No static thresholds! Uses percentile-based adaptive detection:

How it works:

Example History: [100, 110, 95, 105, 98, 102, 110] RPS

P50 (median) = 102 RPS
P95 (95th percentile) = 110 RPS

Trigger HighRPS when: current > P95 + (P95 × hysteresis)
                     current > 110 + (110 × 0.10)
                     current > 121 RPS

Trigger LowRPS when: current < P50 - (P50 × hysteresis)
                    current < 102 - (102 × 0.10)
                    current < 92 RPS

Benefits:

  • ✅ Adapts to each service's traffic patterns
  • ✅ Works for 5 RPS or 5000 RPS services
  • ✅ No manual threshold tuning needed
  • ✅ Hysteresis prevents flapping

What it tracks:

  • Current RPS (real-time)
  • Average RPS (over time window)
  • P50 & P95 percentiles (for adaptive thresholds)
  • Whether service is local or mirrored

Data structure:

type RPSMetrics struct {
    CurrentRPS float64
    AvgRPS     float64
    P50        float64  // Adaptive low threshold
    P95        float64  // Adaptive high threshold
}

Strategy 3: Health Monitoring​

Tracks service health indicators:

What it tracks:

  • Error rate (percentage of 5xx responses)
  • P95 latency (milliseconds)
  • Health status (healthy / degraded / unhealthy)

Data structure:

type HealthMetrics struct {
    ErrorRate  float64  // 0.0 - 1.0
    P95Latency float64  // milliseconds
    IsHealthy  bool
    Status     string   // "healthy", "degraded", "unhealthy"
}

Strategy 4: Cluster Capacity​

Tracks cluster-wide resources and hot nodes:

What it tracks:

  • Total/Free CPU and Memory
  • CPU/Memory utilization (%)
  • Hot nodes (>80% utilization)
  • CanScale flag (>20% free resources)

Data structure:

type ClusterState struct {
    TotalCPU      float64
    FreeCPU       float64
    CPUUtilization float64
    CanScale      bool
    HotNodes      []NodeInfo  // Nodes >80% util
}

Collection Process​

Concurrent Collection Flow​

Efficiency Optimizations​

Batch API Calls:

// OLD: One call per pod (slow!)
for pod := range pods {
    metrics := getPodMetrics(pod)  // 100 pods = 100 calls
}

// NEW: One batch call (fast!)
allMetrics := getAllPodMetrics()  // 100 pods = 1 call

Informers vs Polling:

// OLD: Poll every 30s
ticker := time.NewTicker(30 * time.Second)
for range ticker.C {
    pods := listPods()  // API call every cycle
}

// NEW: Watch with informers
informer.AddEventHandler(cache.ResourceEventHandlerFuncs{
    AddFunc:    onPodAdd,     // Instant updates
    UpdateFunc: onPodUpdate,  // No polling
    DeleteFunc: onPodDelete,
})

Events Emitted​

StressEvent​

Emitted when pod composite stress exceeds threshold:

{
  "type": "stress_high",
  "serviceName": "api",
  "namespace": "default",
  "isLocal": true,
  "stressScore": 0.85,
  "currentRps": 450.0,
  "reason": "High stress: 85.0%",
  "timestamp": "2024-01-15T10:30:00Z"
}

Triggers when:

  • stress_high: Composite score ≥ 0.80 (+ hysteresis)
  • stress_critical: Composite score ≥ 0.90

Orchestrator action: Traffic shifting strategy

HighRPSEvent​

Emitted when mirrored service RPS exceeds adaptive P95 threshold:

{
  "type": "rps_high",
  "serviceName": "payments-mirrored",
  "namespace": "default",
  "isLocal": false,
  "stressScore": 0.0,
  "currentRps": 550.0,
  "reason": "High RPS: 550 req/s (P95: 500)",
  "timestamp": "2024-01-15T10:30:00Z"
}

Triggers when:

  • Service is mirrored (not local)
  • Current RPS > P95 + (P95 × hysteresis)
  • Example: 550 > 500 + (500 × 0.10) = 550 > 550 ✓

Orchestrator action: Deploy service locally

LowRPSEvent​

Emitted when local service RPS drops below adaptive P50 threshold:

{
  "type": "rps_low",
  "serviceName": "old-api",
  "namespace": "default",
  "isLocal": true,
  "stressScore": 0.15,
  "currentRps": 8.0,
  "reason": "Low RPS: 8 req/s (P50: 10)",
  "timestamp": "2024-01-15T10:30:00Z"
}

Triggers when:

  • Service is local (not mirrored)
  • Current RPS < P50 - (P50 × hysteresis)
  • Example: 8 < 10 - (10 × 0.10) = 8 < 9 ✓

Orchestrator action: Mark for deletion

# Wait 5 minutes for history to build up
# Check history size in logs
  1. Services not instrumented:

Symptoms:

  • Monitor OOMKilled
  • Memory usage growing over time

Solutions:

  1. Reduce history samples:

  1. Reduce history window:

Advanced Features​

Adaptive Threshold Calculation​

// Collect RPS history
history := []float64{100, 110, 95, 105, 98, 102, 110}

// Calculate percentiles
P50 := percentile(history, 50)  // 102 RPS
P95 := percentile(history, 95)  // 110 RPS

// Dynamic thresholds adapt to each service
if currentRPS > P95 + (P95 * hysteresis) {
    emitHighRPSEvent()
}

Benefits:

  • No manual tuning per service
  • Handles different scales (5 RPS vs 5000 RPS)
  • Adapts to traffic pattern changes
  • Prevents false positives during spikes

Hot Node Detection​

Identifies nodes under heavy load for proactive management:

// Node with >80% CPU or Memory utilization
type NodeInfo struct {
    NodeName          string
    CPUUtilization    float64  // 0.85 = 85%
    MemoryUtilization float64  // 0.78 = 78%
    PodCount          int      // Number of pods
}

// Use cases:
// 1. Avoid scheduling new pods on hot nodes
// 2. Trigger node autoscaling
// 3. Alert operations team

Event Throttling​

Prevents event storms using cooldown periods:

// Don't emit same event type for same service within 2 minutes
lastEventTime := eventCache["stress_high-api"]
if time.Since(lastEventTime) < 2*time.Minute {
    // Skip event (throttled)
    return
}

// Also applies hysteresis to thresholds
// StressHigh triggers at 0.80
// But only releases at 0.75 (0.80 - 0.05)
// Prevents oscillation

Composite Stress Scoring​

Customize CPU/Memory weights per service type:

service_overrides:
  # CPU-heavy workload (video encoding, ML)
  video-processor:
    cpu_weight: 0.9
    memory_weight: 0.1
  
  # Memory-heavy workload (cache, database)
  redis-cache:
    cpu_weight: 0.3
    memory_weight: 0.7
  
  # Balanced workload (web services)
  web-api:
    cpu_weight: 0.7
    memory_weight: 0.3

Score calculation:

compositeScore = (cpuUsage * cpuWeight) + (memUsage * memWeight)

// Example: Video processor at CPU=0.90, Mem=0.50
score = (0.90 * 0.9) + (0.50 * 0.1) = 0.81 + 0.05 = 0.86 → High stress!

// Example: Redis cache at CPU=0.50, Mem=0.90
score = (0.50 * 0.3) + (0.90 * 0.7) = 0.15 + 0.63 = 0.78 → Medium stress

Health-Based Alerting​

Correlate stress with actual service health:

// Only alert if service is both stressed AND unhealthy
if stress > 0.80 && (errorRate > 0.05 || latency > 1000) {
    // Service is legitimately struggling
    emitCriticalAlert()
} else if stress > 0.80 && errorRate < 0.01 {
    // High stress but service is fine
    // Maybe just handling load well
    emitInfoAlert()
}

Best Practices​

  1. Start with adaptive thresholds - Let the system learn traffic patterns
  2. Monitor collection stats - Check /metrics/stats for success rates
  3. Tune per-service weights - CPU-heavy vs memory-heavy workloads
  4. Set appropriate cooldowns - Balance responsiveness vs event storms
  5. Watch hot nodes - Proactively manage cluster capacity
  6. Use health metrics - Correlate stress with actual service issues
  7. Keep history small - 50 samples sufficient for edge clusters
  8. Alert on collection failures - Critical for system reliability
  9. Test adaptive thresholds - Verify with load testing
  10. Monitor percentile drift - Track how P50/P95 change over time

# Step 2: Keep static as fallback

LOW_RPS_THRESHOLD: "10.0"

# Step 3: Wait for history (10-15 minutes)
# Monitor logs for "calculated percentiles"

# Step 4: Verify adaptive thresholds
curl http://metrics-monitor:8080/metrics/rps?service=your-service
# Check P50 and P95 values are reasonable

# Step 5: (Optional) Remove static thresholds once stable

From v1 to v2 Monitor​

Breaking changes:

  • Event structure simplified (no priority field)
  • New endpoints: /metrics/health, /metrics/stats
  • ConfigMap keys changed (see Configuration section)

Migration steps:

  1. Update ConfigMap:
kubectl apply -f new-defog-config.yaml
  1. Deploy v2 monitor:
kubectl set image deployment/metrics-monitor \
  metrics-monitor=registry.io/defog-metrics-monitor:v2.0.0
  1. Update Orchestrator to handle new event structure
  2. Monitor for errors in first 15 minutes
  3. Verify adaptive thresholds are calculating
