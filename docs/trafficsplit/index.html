Traffic Splits & Management

What is Traffic Management?​

Traffic management is the ability to control how requests flow between services in your system. Instead of requests always going to the same destination, you can:

  • Split traffic between multiple versions of a service
  • Route traffic based on conditions
  • Shift traffic gradually during deployments
  • Distribute load across clusters

Why this matters:

  • ✅ Safe deployments (canary, blue-green)
  • ✅ A/B testing
  • ✅ Load distribution
  • ✅ Failover handling
  • ✅ Multi-cluster routing

Traffic Splits in Linkerd​

Linkerd uses the Service Mesh Interface (SMI) TrafficSplit resource to manage traffic distribution. It's a Kubernetes custom resource that tells Linkerd how to split traffic.

The Basic Concept​

Without traffic splitting, Kubernetes routes requests to all pods equally:

With traffic splitting, you control the distribution:

How It Works​

The flow:

  1. Client calls the original service name (my-app)
  2. Linkerd intercepts the request
  3. TrafficSplit resource defines the routing
  4. Request goes to appropriate backend based on weights
  5. Client receives response (doesn't know which version served it)

TrafficSplit Resource Structure​

Here's what a TrafficSplit looks like:

apiVersion: split.smi-spec.io/v1alpha1
kind: TrafficSplit
metadata:
  name: my-app-split
  namespace: default
spec:
  # The service clients call
  service: my-app
  
  # Where traffic actually goes
  backends:
  - service: my-app-v1
    weight: 900    # 90% of traffic
  - service: my-app-v2
    weight: 100    # 10% of traffic

Key fields:

FieldPurposeExample
serviceThe "virtual" service clients callmy-app
backendsList of actual services receiving trafficmy-app-v1, my-app-v2
weightProportion of traffic (out of 1000)900 = 90%, 100 = 10%
Weight System

Weights are out of 1000, not 100. This allows fine-grained control:

  • 995 = 99.5%
  • 5 = 0.5%

Setting Up Traffic Splits​

Step 1: Deploy Multiple Service Versions​

Create your deployments and services:

# Version 1
apiVersion: v1
kind: Service
metadata:
  name: my-app-v1
spec:
  selector:
    app: my-app
    version: v1
  ports:
  - port: 8080
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app-v1
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app
      version: v1
  template:
    metadata:
      labels:
        app: my-app
        version: v1
    spec:
      containers:
      - name: app
        image: my-app:v1
        ports:
        - containerPort: 8080
# Version 2  
apiVersion: v1
kind: Service
metadata:
  name: my-app-v2
spec:
  selector:
    app: my-app
    version: v2
  ports:
  - port: 8080
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app-v2
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app
      version: v2
  template:
    metadata:
      labels:
        app: my-app
        version: v2
    spec:
      containers:
      - name: app
        image: my-app:v2
        ports:
        - containerPort: 8080

Step 2: Create the Apex Service​

The "apex" service is what clients call:

apiVersion: v1
kind: Service
metadata:
  name: my-app
spec:
  # No selector! TrafficSplit handles routing
  ports:
  - port: 8080
Important

The apex service should have no selector. Linkerd uses TrafficSplit to route instead.

Step 3: Create the TrafficSplit​

apiVersion: split.smi-spec.io/v1alpha1
kind: TrafficSplit
metadata:
  name: my-app-split
spec:
  service: my-app
  backends:
  - service: my-app-v1
    weight: 1000   # Start with 100% on v1
  - service: my-app-v2
    weight: 0      # 0% on v2

Step 4: Gradually Shift Traffic​

Update the weights to shift traffic:

# 90% v1, 10% v2
kubectl patch trafficsplit my-app-split --type merge -p '
spec:
  backends:
  - service: my-app-v1
    weight: 900
  - service: my-app-v2
    weight: 100
'

# 50% v1, 50% v2
kubectl patch trafficsplit my-app-split --type merge -p '
spec:
  backends:
  - service: my-app-v1
    weight: 500
  - service: my-app-v2
    weight: 500
'

# 0% v1, 100% v2 (complete migration)
kubectl patch trafficsplit my-app-split --type merge -p '
spec:
  backends:
  - service: my-app-v1
    weight: 0
  - service: my-app-v2
    weight: 1000
'

Common Use Cases​

1. Canary Deployments​

Gradually roll out a new version to minimize risk.

TrafficSplit progression:

  • Step 1: Deploy
  • Step 2: Test (10%)
  • Step 3: Expand (50%)
  • Step 4: Complete (100%)
# Deploy v2, send no traffic yet
backends:
- service: my-app-v1
  weight: 1000
- service: my-app-v2
  weight: 0
# Send 10% of traffic to v2
backends:
- service: my-app-v1
  weight: 900
- service: my-app-v2
  weight: 100

Monitor error rates, latency, resource usage

# If v2 looks good, increase to 50%
backends:
- service: my-app-v1
  weight: 500
- service: my-app-v2
  weight: 500
# Full migration to v2
backends:
- service: my-app-v1
  weight: 0
- service: my-app-v2
  weight: 1000

Now safe to delete v1

2. Blue-Green Deployments​

Keep two complete environments and switch instantly.

Switch instantly:

# Before: All traffic to blue
backends:
- service: my-app-blue
  weight: 1000
- service: my-app-green
  weight: 0

# After: Instant switch to green  
backends:
- service: my-app-blue
  weight: 0
- service: my-app-green
  weight: 1000

Benefits:

  • Instant rollback if issues found
  • Zero downtime
  • Full environment testing before switch

3. A/B Testing​

Send different traffic to different versions for testing.

# 50/50 split for A/B testing
apiVersion: split.smi-spec.io/v1alpha1
kind: TrafficSplit
metadata:
  name: frontend-ab-test
spec:
  service: frontend
  backends:
  - service: frontend-version-a
    weight: 500
  - service: frontend-version-b
    weight: 500

Use case:

  • Test UI changes
  • Compare performance
  • Validate features
  • Gather metrics

4. Multi-Cluster Failover​

Route traffic between local and remote clusters.

apiVersion: split.smi-spec.io/v1alpha1
kind: TrafficSplit
metadata:
  name: api-failover
spec:
  service: api
  backends:
  - service: api-local
    weight: 1000        # Prefer local
  - service: api-backup-cluster
    weight: 0           # Backup only

When to switch:

  • Local cluster degraded
  • High latency detected
  • Resource constraints
  • Planned maintenance

Switch to backup:

kubectl patch trafficsplit api-failover --type merge -p '
spec:
  backends:
  - service: api-local
    weight: 0
  - service: api-backup-cluster
    weight: 1000
'

Traffic Splits with Mirrored Services​

In multi-cluster setups, TrafficSplits work with mirrored services.

Scenario​

You have:

  • Cluster A (Edge): Wants to call hello service
  • Cluster B (Fog): Has hello service exported

Linkerd creates:

  • In Cluster A: hello-clusterB (mirrored service)

Using TrafficSplit with Mirrors​

apiVersion: split.smi-spec.io/v1alpha1
kind: TrafficSplit
metadata:
  name: hello-split
  namespace: default
spec:
  service: hello
  backends:
  - service: hello-local      # Local instance if exists
    weight: 800               # Prefer local
  - service: hello-clusterB   # Mirrored from Cluster B
    weight: 200               # Use remote for 20%

Traffic flow:

Use cases:

  • Load distribution across clusters
  • Gradual migration between clusters
  • Failover to remote cluster
  • Geographic routing

Advanced Patterns​

Weighted Round-Robin Across Three Versions​

apiVersion: split.smi-spec.io/v1alpha1
kind: TrafficSplit
metadata:
  name: multi-version-split
spec:
  service: api
  backends:
  - service: api-v1
    weight: 700    # 70% - stable version
  - service: api-v2
    weight: 250    # 25% - testing version
  - service: api-v3
    weight: 50     # 5% - experimental version

Cluster Locality Preference​

# Prefer local cluster, fallback to remote
apiVersion: split.smi-spec.io/v1alpha1
kind: TrafficSplit
metadata:
  name: locality-split
spec:
  service: database
  backends:
  - service: database-local
    weight: 900        # 90% local
  - service: database-cloud
    weight: 100        # 10% remote (for testing)

Namespace-Based Splits​

# Route based on where request originates
apiVersion: split.smi-spec.io/v1alpha1
kind: TrafficSplit
metadata:
  name: env-split
  namespace: production
spec:
  service: shared-api
  backends:
  - service: api-prod
    weight: 1000
---
apiVersion: split.smi-spec.io/v1alpha1
kind: TrafficSplit
metadata:
  name: env-split
  namespace: staging
spec:
  service: shared-api
  backends:
  - service: api-staging
    weight: 1000

Monitoring Traffic Splits​

Using Linkerd Viz​

# View traffic distribution
linkerd viz stat trafficsplit

# Watch specific split
linkerd viz stat trafficsplit/my-app-split

# View live traffic
linkerd viz tap deploy/my-app-v1
linkerd viz tap deploy/my-app-v2

Metrics to Watch​

Success Rate:

linkerd viz stat deploy/my-app-v1 deploy/my-app-v2

Latency:

linkerd viz stat deploy/my-app-v1 deploy/my-app-v2 -o json | \
  jq '.rows[] | {name: .resource, p50: .stats.latency_ms_p50}'

Request Volume:

linkerd viz edges deployment

Dashboard View​

linkerd viz dashboard

In the dashboard:

  1. Navigate to Deployments
  2. Compare metrics between versions
  3. Watch real-time traffic distribution
  4. Monitor success rates and latency

Best Practices​

1. Start Conservative​

# Start with small percentage
backends:
- service: stable
  weight: 990   # 99%
- service: new
  weight: 10    # 1%

2. Monitor Before Increasing​

Wait and watch metrics before increasing traffic:

  • Success rate stays high
  • Latency doesn't increase
  • No new errors
  • Resource usage acceptable

3. Use Meaningful Names​

# Good
service: user-api
backends:
- service: user-api-stable
- service: user-api-canary

# Bad
service: api
backends:
- service: api1
- service: api2

4. Document Your Splits​

apiVersion: split.smi-spec.io/v1alpha1
kind: TrafficSplit
metadata:
  name: payment-canary
  annotations:
    purpose: "Canary deployment for payment service v2.1"
    owner: "payments-team"
    created: "2024-01-15"
spec:
  service: payment
  backends:
  - service: payment-v2-0
    weight: 950
  - service: payment-v2-1
    weight: 50

5. Have a Rollback Plan​

# Save current state
kubectl get trafficsplit my-app-split -o yaml > backup.yaml

# Test new weights
kubectl apply -f new-split.yaml

# Rollback if needed
kubectl apply -f backup.yaml

6. Automate Gradual Rollouts​

#!/bin/bash
# gradual-rollout.sh

SPLIT_NAME="my-app-split"
STEPS=(0 100 250 500 750 900 1000)

for weight in "${STEPS[@]}"; do
  echo "Shifting to $weight/1000..."
  
  kubectl patch trafficsplit $SPLIT_NAME --type merge -p "
  spec:
    backends:
    - service: my-app-v1
      weight: $((1000 - weight))
    - service: my-app-v2
      weight: $weight
  "
  
  echo "Waiting 5 minutes for metrics..."
  sleep 300
  
  # Check metrics here
  # If bad, rollback and exit
done

echo "Rollout complete!"

Troubleshooting​

Traffic Not Splitting​

Check TrafficSplit exists:

kubectl get trafficsplit
kubectl describe trafficsplit my-app-split

Verify services exist:

kubectl get services my-app my-app-v1 my-app-v2

Check apex service has no selector:

kubectl get service my-app -o yaml | grep selector
# Should return nothing or "selector: {}"

All Traffic Going to One Backend​

Check weights:

kubectl get trafficsplit my-app-split -o yaml

Verify pods are running:

kubectl get pods -l app=my-app,version=v2

Check Linkerd proxy:

linkerd viz stat deploy/my-app-v2

Metrics Not Showing Split​

Ensure Linkerd injection:

kubectl get pods -l app=my-app -o jsonpath='{.items[*].spec.containers[*].name}'
# Should include "linkerd-proxy"

Check SMI extension:

kubectl get crds | grep split
# Should show: trafficsplits.split.smi-spec.io

Real-World Example: Complete Canary Deployment​

Here's a complete example deploying a canary update:

# 1. Original deployment (v1)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: webapp-v1
spec:
  replicas: 5
  selector:
    matchLabels:
      app: webapp
      version: v1
  template:
    metadata:
      labels:
        app: webapp
        version: v1
      annotations:
        linkerd.io/inject: enabled
    spec:
      containers:
      - name: webapp
        image: webapp:1.0.0
        ports:
        - containerPort: 8080
---
apiVersion: v1
kind: Service
metadata:
  name: webapp-v1
spec:
  selector:
    app: webapp
    version: v1
  ports:
  - port: 80
    targetPort: 8080
---
# 2. New canary deployment (v2)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: webapp-v2
spec:
  replicas: 2
  selector:
    matchLabels:
      app: webapp
      version: v2
  template:
    metadata:
      labels:
        app: webapp
        version: v2
      annotations:
        linkerd.io/inject: enabled
    spec:
      containers:
      - name: webapp
        image: webapp:2.0.0
        ports:
        - containerPort: 8080
---
apiVersion: v1
kind: Service
metadata:
  name: webapp-v2
spec:
  selector:
    app: webapp
    version: v2
  ports:
  - port: 80
    targetPort: 8080
---
# 3. Apex service (no selector)
apiVersion: v1
kind: Service
metadata:
  name: webapp
spec:
  ports:
  - port: 80
    targetPort: 8080
---
# 4. TrafficSplit (start with 5% canary)
apiVersion: split.smi-spec.io/v1alpha1
kind: TrafficSplit
metadata:
  name: webapp-canary
spec:
  service: webapp
  backends:
  - service: webapp-v1
    weight: 950
  - service: webapp-v2
    weight: 50

Deployment process:

# 1. Deploy everything
kubectl apply -f webapp-canary.yaml

# 2. Monitor for 10 minutes
linkerd viz stat deploy/webapp-v1 deploy/webapp-v2

# 3. If good, increase to 25%
kubectl patch trafficsplit webapp-canary --type merge -p '
spec:
  backends:
  - service: webapp-v1
    weight: 750
  - service: webapp-v2
    weight: 250
'

# 4. Monitor again, then 50%
kubectl patch trafficsplit webapp-canary --type merge -p '
spec:
  backends:
  - service: webapp-v1
    weight: 500
  - service: webapp-v2
    weight: 500
'

# 5. Finally, 100%
kubectl patch trafficsplit webapp-canary --type merge -p '
spec:
  backends:
  - service: webapp-v1
    weight: 0
  - service: webapp-v2
    weight: 1000
'

# 6. Clean up old version
kubectl delete deployment webapp-v1
kubectl delete service webapp-v1
