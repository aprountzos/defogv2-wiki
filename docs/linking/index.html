Linking Clusters

Overview​

Now that Linkerd is installed on all clusters, we'll connect them together to enable secure cross-cluster communication.

What we'll create:

  • Link from Bot to Mid - Bot can access Mid services
  • Link from Mid to Top - Mid can access Top services

This creates a hierarchical architecture mimicking real cloud-fog-edge topology.

Prerequisites​

  • ✅ All three clusters running
  • ✅ Linkerd installed on all clusters
  • ✅ MetalLB configured with gateway IPs assigned
  • ✅ All linkerd check and linkerd multicluster check passing

Understanding Links​

What is a Link?​

A Link is a Linkerd resource that:

  1. Establishes a secure connection to another cluster
  2. Enables service mirroring from that cluster
  3. Creates a gateway tunnel for cross-cluster traffic

Link Direction​

Links are directional:

  • Source cluster → creates Link → Target cluster
  • Source can discover services from Target
  • Target doesn't automatically discover services from Source

Our setup:

Bot (source) → Link → Mid (target)
Mid (source) → Link → Top (target)

Result:

  • Bot can call services in Mid and Top (via Mid)
  • Mid can call services in Top
  • Top cannot call services in Mid or Bot (unless we create reverse links)

Step 1: Link Bot to Mid​

We'll connect the Bot cluster to the Mid cluster.

Step 1.1: Get Mid Cluster Information​

First, gather information about the Mid cluster:

# Get Mid gateway IP
MID_GATEWAY_IP=$(kubectl get svc -n linkerd-multicluster linkerd-gateway \
  --context k3d-mid \
  -o jsonpath='{.status.loadBalancer.ingress[0].ip}')

echo "Mid Gateway IP: $MID_GATEWAY_IP"

# Get Mid API server address
MID_SERVER_IP=$(docker inspect k3d-mid-serverlb \
  --format '{{range .NetworkSettings.Networks}}{{.IPAddress}}{{end}}')

MID_API="https://${MID_SERVER_IP}:6443"

echo "Mid API Server: $MID_API"

Save these values! You'll need them in the next steps.

Example output:

Mid Gateway IP: 172.18.0.210
Mid API Server: https://172.18.0.4:6443

Step 1.2: Generate Link Manifest​

Generate the Link resource that Bot will use to connect to Mid:

linkerd multicluster link \
  --context k3d-mid \
  --cluster-name mid \
  --api-server-address $MID_API \
  --gateway-addresses ${MID_GATEWAY_IP}:4143 \
  --selector to-export=true > link-bot-to-mid.yaml

What each flag does:

FlagPurposeValue
--cluster-nameName for the target clustermid
--api-server-addressAPI server of target clusterMid's API address
--gateway-addressesGateway IP and portMid's gateway IP:4143
--selectorLabel selector for exported servicesto-export=true
Service Selector

Only services with the label to-export=true in Mid will be mirrored to Bot. This gives you control over what's shared.

Step 1.3: Inspect the Link​

Let's see what was generated:

cat link-bot-to-mid.yaml

You'll see:

  • Link resource - Connection configuration
  • Secret - Credentials to access Mid cluster
  • ServiceAccount - For the service mirror controller

Step 1.4: Apply the Link​

Apply the Link in the Bot cluster:

kubectl apply --context k3d-bot -f link-bot-to-mid.yaml

Expected output:

secret/cluster-credentials-mid created
link.multicluster.linkerd.io/mid created
clusterrole.rbac.authorization.k8s.io/linkerd-service-mirror-access-local-resources-mid created
clusterrolebinding.rbac.authorization.k8s.io/linkerd-service-mirror-access-local-resources-mid created
role.rbac.authorization.k8s.io/linkerd-service-mirror-read-remote-creds-mid created
rolebinding.rbac.authorization.k8s.io/linkerd-service-mirror-read-remote-creds-mid created
serviceaccount/linkerd-service-mirror-mid created
deployment.apps/linkerd-service-mirror-mid created
service/probe-gateway-mid created

Step 1.5: Verify the Link​

Check the Link status:

# View Link resource
kubectl get link --context k3d-bot -n linkerd-multicluster

# Check if gateway is alive
linkerd multicluster gateways --context k3d-bot

Expected output:

CLUSTER   ALIVE    NUM_SVC   LATENCY
mid       True     0         1ms

✅ ALIVE should be True!

Step 1.6: Check Service Mirror Controller​

kubectl get pods -n linkerd-multicluster --context k3d-bot

You should see a new pod:

NAME                                      READY   STATUS    RESTARTS   AGE
linkerd-gateway-xxx                       2/2     Running   0          10m
linkerd-service-mirror-mid-xxx            2/2     Running   0          1m

Step 1.7: View Logs (Optional)​

See what the service mirror is doing:

kubectl logs -n linkerd-multicluster \
  -l app=linkerd-service-mirror \
  -c controller \
  --context k3d-bot \
  --tail 20

Step 2: Link Mid to Top​

Now connect the Mid cluster to the Top cluster.

Step 2.1: Get Top Cluster Information​

# Get Top gateway IP
TOP_GATEWAY_IP=$(kubectl get svc -n linkerd-multicluster linkerd-gateway \
  --context k3d-top \
  -o jsonpath='{.status.loadBalancer.ingress[0].ip}')

echo "Top Gateway IP: $TOP_GATEWAY_IP"

# Get Top API server address
TOP_SERVER_IP=$(docker inspect k3d-top-serverlb \
  --format '{{range .NetworkSettings.Networks}}{{.IPAddress}}{{end}}')

TOP_API="https://${TOP_SERVER_IP}:6443"

echo "Top API Server: $TOP_API"

Step 2.2: Generate Link Manifest​

linkerd multicluster link \
  --context k3d-top \
  --cluster-name top \
  --api-server-address $TOP_API \
  --gateway-addresses ${TOP_GATEWAY_IP}:4143 \
  --selector to-export=true > link-mid-to-top.yaml

Step 2.3: Apply the Link​

kubectl apply --context k3d-mid -f link-mid-to-top.yaml

Step 2.4: Verify the Link​

# Check Link
kubectl get link --context k3d-mid -n linkerd-multicluster

# Check gateway status
linkerd multicluster gateways --context k3d-mid

Expected:

CLUSTER   ALIVE    NUM_SVC   LATENCY
top       True     0         1ms

Step 2.5: Verify Controller​

kubectl get pods -n linkerd-multicluster --context k3d-mid

Should show:

linkerd-gateway-xxx                       2/2     Running   0          10m
linkerd-service-mirror-top-xxx            2/2     Running   0          1m

Step 3: Verify All Links​

Check Link Status​

  • Bot Cluster
  • Mid Cluster
  • Top Cluster
linkerd multicluster check --context k3d-bot
linkerd multicluster gateways --context k3d-bot

Should show link to mid as ALIVE.

linkerd multicluster check --context k3d-mid
linkerd multicluster gateways --context k3d-mid

Should show link to top as ALIVE.

linkerd multicluster check --context k3d-top
linkerd multicluster gateways --context k3d-top

Should show no links (Top is the apex).

Testing the Links​

Create a Test Service in Top​

# Create namespace
kubectl create namespace test --context k3d-top

# Annotate for Linkerd injection
kubectl annotate namespace test linkerd.io/inject=enabled --context k3d-top

# Deploy hello service
kubectl apply --context k3d-top -n test -f - <<EOF
apiVersion: apps/v1
kind: Deployment
metadata:
  name: hello
spec:
  replicas: 1
  selector:
    matchLabels:
      app: hello
  template:
    metadata:
      labels:
        app: hello
    spec:
      containers:
      - name: hello
        image: hashicorp/http-echo
        args:
        - "-text=Hello from Top cluster!"
        - "-listen=:8080"
        ports:
        - containerPort: 8080
---
apiVersion: v1
kind: Service
metadata:
  name: hello
  labels:
    to-export: "true"
spec:
  selector:
    app: hello
  ports:
  - name: http
    port: 8080
    targetPort: 8080
EOF
Export Label

Notice the to-export: "true" label on the Service. This is required for mirroring!

Wait for Service to Be Ready​

kubectl wait --for=condition=ready pod \
  -l app=hello \
  -n test \
  --context k3d-top \
  --timeout=60s

Check for Mirrored Service in Mid​

# Should see hello-top service appear
kubectl get services -n test --context k3d-mid --watch

Wait for hello-top to appear (may take 30 seconds):

NAME        TYPE        CLUSTER-IP      EXTERNAL-IP   PORT(S)    AGE
hello-top   ClusterIP   10.43.xxx.xxx   <none>        8080/TCP   10s

Press Ctrl+C when you see it.

Verify Gateway Status​

linkerd multicluster gateways --context k3d-mid

Should now show:

CLUSTER   ALIVE    NUM_SVC   LATENCY
top       True     1         1ms

NUM_SVC changed from 0 to 1! ✅

Check for Mirrored Service in Bot​

The service should also appear in Bot (via Mid):

kubectl get services -n test --context k3d-bot --watch

Wait for hello-top to appear.

Testing Connectivity​

Deploy Test Pod in Bot​

# Create namespace
kubectl create namespace test --context k3d-bot

# Annotate for injection
kubectl annotate namespace test linkerd.io/inject=enabled --context k3d-bot

# Deploy curl pod
kubectl apply --context k3d-bot -n test -f - <<EOF
apiVersion: v1
kind: Pod
metadata:
  name: curl
spec:
  containers:
  - name: curl
    image: curlimages/curl
    command: ["sleep", "infinity"]
EOF

Wait for Pod​

kubectl wait --for=condition=ready pod/curl \
  -n test \
  --context k3d-bot \
  --timeout=60s

Test Connection​

# Test calling hello-top from Bot
kubectl exec -it curl -n test --context k3d-bot -- \
  curl -s http://hello-top:8080

Expected output:

Hello from Top cluster!

🎉 Success! Bot is calling a service in Top through the secure gateway tunnel!

View Traffic in Linkerd Viz​

# Open dashboard
linkerd viz dashboard --context k3d-bot

Navigate to the test namespace and watch the traffic flow.

Advanced: Creating a Service in Mid​

Let's create a service in Mid and test the full chain.

Deploy Service in Mid​

# Create namespace if needed
kubectl create namespace test --context k3d-mid --dry-run=client -o yaml | \
  kubectl apply --context k3d-mid -f -

# Annotate
kubectl annotate namespace test linkerd.io/inject=enabled --context k3d-mid --overwrite

# Deploy service
kubectl apply --context k3d-mid -n test -f - <<EOF
apiVersion: apps/v1
kind: Deployment
metadata:
  name: hello
spec:
  replicas: 1
  selector:
    matchLabels:
      app: hello
  template:
    metadata:
      labels:
        app: hello
    spec:
      containers:
      - name: hello
        image: hashicorp/http-echo
        args:
        - "-text=Hello from Mid cluster!"
        - "-listen=:8080"
        ports:
        - containerPort: 8080
---
apiVersion: v1
kind: Service
metadata:
  name: hello
  labels:
    to-export: "true"
spec:
  selector:
    app: hello
  ports:
  - name: http
    port: 8080
    targetPort: 8080
EOF

Check Mirroring in Bot​

kubectl get services -n test --context k3d-bot

Should now see:

NAME        TYPE        CLUSTER-IP      EXTERNAL-IP   PORT(S)    AGE
hello-mid   ClusterIP   10.43.xxx.xxx   <none>        8080/TCP   10s
hello-top   ClusterIP   10.43.xxx.xxx   <none>        8080/TCP   5m

Test Both Services​

# Call Mid service
kubectl exec -it curl -n test --context k3d-bot -- \
  curl -s http://hello-mid:8080

# Call Top service
kubectl exec -it curl -n test --context k3d-bot -- \
  curl -s http://hello-top:8080

Expected:

Hello from Mid cluster!
Hello from Top cluster!

Perfect! Bot can now reach services in both Mid and Top! 🎉

Troubleshooting​

Link Shows as Not Alive​

Check gateway is accessible:

# From source cluster, test gateway connectivity
kubectl exec -n linkerd-multicluster \
  -it $(kubectl get pod -n linkerd-multicluster -l app=linkerd-gateway --context k3d-bot -o jsonpath='{.items[0].metadata.name}') \
  --context k3d-bot \
  -c nginx -- curl -vk https://MID_GATEWAY_IP:4143

Check gateway pod:

kubectl get pods -n linkerd-multicluster --context k3d-mid
kubectl logs -n linkerd-multicluster -l app=linkerd-gateway --context k3d-mid

Verify MetalLB:

kubectl get svc -n linkerd-multicluster linkerd-gateway --context k3d-mid

Gateway should have EXTERNAL-IP assigned.

Services Not Being Mirrored​

Check service has export label:

kubectl get svc hello -n test --context k3d-top -o yaml | grep to-export

Should show:

labels:
  to-export: "true"

Check service mirror logs:

kubectl logs -n linkerd-multicluster \
  -l app=linkerd-service-mirror \
  -c controller \
  --context k3d-bot \
  --tail 50

Verify Link selector:

kubectl get link -n linkerd-multicluster --context k3d-bot -o yaml

Check selector matches your service labels.

Can't Connect to Mirrored Service​

Check service exists:

kubectl get svc -n test --context k3d-bot

Check endpoints:

kubectl get endpoints -n test --context k3d-bot

Test DNS resolution:

kubectl exec -it curl -n test --context k3d-bot -- \
  nslookup hello-mid.test.svc.cluster.local

Check Linkerd proxy:

kubectl exec -it curl -n test --context k3d-bot -c linkerd-proxy -- \
  wget -qO- http://localhost:4191/metrics | grep hello-mid

Understanding the Complete Architecture​

Current Setup​

Service Discovery​

From Bot's perspective:

# List all services
kubectl get svc -A --context k3d-bot

# Services Bot can call:
# - Local services (in Bot)
# - hello-mid (in Mid, via gateway)
# - hello-top (in Top, via Mid's gateway)

Managing Links​

Viewing Link Details​

# Get Link details
kubectl get link mid -n linkerd-multicluster --context k3d-bot -o yaml

# Check service mirror deployment
kubectl get deployment -n linkerd-multicluster --context k3d-bot

# View gateway status
linkerd multicluster gateways --context k3d-bot

Updating a Link​

If gateway IP changes:

# Delete old link
kubectl delete link mid -n linkerd-multicluster --context k3d-bot

# Regenerate and apply
linkerd multicluster link \
  --context k3d-mid \
  --cluster-name mid \
  --api-server-address $NEW_MID_API \
  --gateway-addresses ${NEW_MID_GATEWAY_IP}:4143 \
  --selector to-export=true | \
  kubectl apply --context k3d-bot -f -

Deleting a Link​

# Delete link resource
kubectl delete link mid -n linkerd-multicluster --context k3d-bot

# This will:
# - Stop service mirroring
# - Remove mirrored services
# - Delete service mirror controller

Quick Reference​

Link Creation Command​

# General format
linkerd multicluster link \
  --context <TARGET_CONTEXT> \
  --cluster-name <NAME_FOR_TARGET> \
  --api-server-address <TARGET_API> \
  --gateway-addresses <TARGET_GATEWAY_IP>:4143 \
  --selector <LABEL_SELECTOR> | \
  kubectl apply --context <SOURCE_CONTEXT> -f -

Verification Commands​

# Check links
kubectl get links -A --context <CLUSTER>

# Check gateways
linkerd multicluster gateways --context <CLUSTER>

# Check mirrored services
kubectl get svc -A --context <CLUSTER> | grep -E "NAME|-(top|mid|bot)"

# Test connectivity
kubectl exec -it <POD> -n <NAMESPACE> --context <CLUSTER> -- \
  curl http://<SERVICE>-<CLUSTER>:<PORT>
