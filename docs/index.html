Defog v2

Intelligent Multi-Cluster Workload Management for Kubernetes

Defog v2 automatically distributes services across Kubernetes clusters using Linkerd multi-cluster, combining stress-based traffic management with RPS-based service placement.

What is Defog v2?â€‹

Defog v2 is a distributed system that helps you:

  • Automatically balance load across clusters when pods are stressed
  • Deploy popular services locally based on request patterns
  • Optimize resource usage by removing low-traffic services
  • Enable seamless failover when services become unavailable
  • Coordinate peer clusters without centralized control

Key Featuresâ€‹

ðŸš€ Dual-Strategy Approachâ€‹

Stress Strategy - Immediate Relief

  • Monitors pod CPU/RAM usage
  • Shifts traffic to mirrored services
  • Response time: 5-10 seconds
  • Automatically reverses when stress reduces

RPS Strategy - Long-term Optimization

  • Tracks requests per second (RPS)
  • Deploys popular mirrored services locally
  • Removes low-traffic services (LFU algorithm)
  • Coordinates with peer clusters when needed

ðŸŽ¯ Autonomous Operationâ€‹

Each cluster runs independently:

  • No centralized controller
  • Peer-to-peer communication only when necessary
  • Each cluster makes its own decisions
  • Survives network partitions

ðŸ”„ Seamless Integrationâ€‹

Built on standard Kubernetes tools:

  • Linkerd for service mirroring
  • TrafficSplit CRD for traffic management
  • Kubernetes API for deployments
  • Prometheus for metrics (optional)

How It Worksâ€‹

Quick Exampleâ€‹

Scenario: Your API service in Cluster A is overloaded

Result:

  1. Immediate relief via traffic shifting (10 seconds)
  2. Long-term fix via local deployment (60 seconds)
  3. Optimal state: All traffic local with sufficient capacity

Architectureâ€‹

Defog v2 consists of 5 independent microservices:

Use Casesâ€‹

1. Multi-Region High Availabilityâ€‹

Challenge: Maintain low latency while ensuring resilience

Solution:

  • Serve users from nearest cluster
  • Automatically failover to remote cluster if needed
  • Deploy popular services in both regions

2. Burst Traffic Handlingâ€‹

Challenge: Handle traffic spikes without over-provisioning

Solution:

  • Normal load: All traffic local
  • Spike detected: Shift 30-50% to peer cluster
  • Spike ends: Shift traffic back
  • No manual intervention needed

3. Cost Optimizationâ€‹

Challenge: Minimize cluster costs while maintaining performance

Solution:

  • Deploy only frequently-used services
  • Remove services with < 10 RPS
  • Coordinate with cheaper peer clusters for overflow

4. Progressive Service Migrationâ€‹

Challenge: Migrate services between clusters safely

Solution:

  • Deploy service in new cluster
  • Gradually shift traffic (10%, 30%, 50%, 100%)
  • Monitor and rollback if needed
  • Remove from old cluster when stable

Quick Startâ€‹

Prerequisitesâ€‹

# Verify prerequisites
kubectl version --short        # Kubernetes 1.24+
linkerd version               # Linkerd 2.12+
linkerd multicluster check    # Multi-cluster configured

1. Install Defog v2â€‹

# Clone repository
git clone https://github.com/your-org/defog-v2.git
cd defog-v2

# Create namespace
kubectl create namespace defog-system

# Apply configuration
kubectl apply -f deployments/kubernetes/

2. Configure Peer Clustersâ€‹

# config/peer-clusters.yaml
apiVersion: v1
kind: Secret
metadata:
  name: peer-cluster-config
  namespace: defog-system
stringData:
  clusters.json: |
    [{
      "id": "cluster-b",
      "apiEndpoint": "https://cluster-b-executor.example.com",
      "region": "us-east-1",
      "token": "your-secure-token"
    }]
kubectl apply -f config/peer-clusters.yaml

3. Verify Installationâ€‹

# Check pods
kubectl get pods -n defog-system

# Should see 5 services running:
# - discovery-controller
# - resource-monitor
# - scheduler
# - orchestrator
# - executor

# Check logs
kubectl logs -n defog-system deployment/orchestrator

4. Watch It Workâ€‹

# Generate load on a service
kubectl run load-generator --image=busybox --restart=Never -- \
  sh -c "while true; do wget -q -O- http://api-service; done"

# Watch metrics
kubectl logs -f -n defog-system deployment/resource-monitor

# Watch traffic splits
kubectl get trafficsplit -w

Configuration Overviewâ€‹

Key settings you'll want to adjust:

# Stress thresholds (when to shift traffic)
STRESS_HIGH: "0.80"           # 80% CPU/RAM
STRESS_CRITICAL: "0.90"       # 90% CPU/RAM

# RPS thresholds (when to deploy/evict)
HIGH_RPS_THRESHOLD: "300.0"   # Deploy if mirrored > 300 RPS
LOW_RPS_THRESHOLD: "10.0"     # Consider deletion if < 10 RPS
MIN_RPS_TO_KEEP: "50.0"       # Never delete if > 50 RPS

# Capacity limits
MAX_MIRRORED_SERVICES: "10"   # Max foreign workloads
MAX_FOREIGN_CPU: "2000m"      # Max 2 cores for foreign services

# Timing
CHECK_INTERVAL: "30s"         # How often to check metrics

See Configuration Guide for complete details.

Design Philosophyâ€‹

1. Autonomous Clustersâ€‹

Each cluster operates independently:

  • Makes its own decisions
  • Communicates with peers only when necessary
  • Survives network partitions
  • No single point of failure

2. Gradual Changesâ€‹

All changes happen incrementally:

  • Traffic shifts by 10% increments
  • Services deployed with ramp-up period
  • Automatic rollback if issues detected
  • Minimize user impact

3. Safety Firstâ€‹

Multiple safety mechanisms:

  • Minimum RPS threshold prevents critical service deletion
  • Capacity limits prevent cluster overload
  • Health checks before peer communication
  • Audit logging for all actions

4. Observableâ€‹

Everything is visible:

  • Detailed logging at each step
  • Prometheus metrics for monitoring
  • API endpoints for inspection
  • Clear event trail
