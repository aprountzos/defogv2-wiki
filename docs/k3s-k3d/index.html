K3s & K3d

What is Kubernetes?​

Before diving into K3s and K3d, let's quickly review what Kubernetes is.

Kubernetes (K8s) is a container orchestration platform that automates deployment, scaling, and management of containerized applications. Think of it as an operating system for your cluster.

What Kubernetes does:

  • Runs your containers across multiple machines
  • Automatically restarts failed containers
  • Scales applications up and down
  • Manages networking between containers
  • Handles load balancing
  • Stores configuration and secrets

The Problem with Standard Kubernetes​

Full Kubernetes is powerful but has challenges:

  • Heavy: Requires significant resources (4GB+ RAM minimum)
  • Complex: Many components and dependencies
  • Slow: Takes minutes to start up
  • Overkill: Too much for development, testing, or edge devices

For cloud-fog-edge architectures, we need something lighter, especially for fog and edge layers.

What is K3s?​

K3s is a lightweight Kubernetes distribution built by Rancher Labs (now part of SUSE). It's certified Kubernetes, meaning it passes all conformance tests and works exactly like standard Kubernetes.

The Name

"K3s" is a play on "K8s" (Kubernetes). The "5 less than 8" implies it's lighter/smaller.

Why K3s is Perfect for Our Use Case​

Key Features:

  • Single Binary: Everything packaged in one ~60MB file
  • Low Resource Usage: Runs in 512MB RAM
  • Fast Startup: Cluster ready in seconds
  • ARM Support: Perfect for Raspberry Pi and edge devices
  • Batteries Included: Comes with essential components built-in
  • Production Ready: Used by major companies in production

What Makes K3s Lightweight?​

K3s achieves its small size by:

  1. Removing Cloud Provider Code: No AWS, Azure, GCP specific code
  2. Replacing etcd: Uses SQLite by default (or can use PostgreSQL/MySQL)
  3. Combining Binaries: All components in one executable
  4. Removing Legacy Features: Only modern, necessary features included
  5. Optimized Storage: Built-in local storage provider

K3s Architecture​

K3s has two types of nodes:

Server Node (Control Plane):

  • Runs the Kubernetes API server
  • Manages cluster state
  • Schedules workloads
  • Can also run workloads

Agent Node (Worker):

  • Runs workloads only
  • Reports back to server
  • Lighter than server node

For our use case, we'll use single-node clusters (server only) to simulate different infrastructure tiers.

What is K3d?​

K3d is a tool that runs K3s clusters in Docker containers. It makes creating and managing multiple Kubernetes clusters trivially easy.

Why K3d?​

Without K3d, setting up multiple clusters requires:

  • Multiple VMs or physical machines
  • Complex networking configuration
  • Significant resources
  • Time-consuming setup

With K3d, you get:

  • Multiple clusters on one laptop
  • Setup in seconds
  • Easy cleanup
  • Perfect for development and testing

K3d vs K3s​

AspectK3sK3d
What it isLightweight Kubernetes distributionTool to run K3s in Docker
Runs onVirtual machines, bare metal, containersDocker containers only
Use caseProduction and developmentDevelopment and testing
SetupManual installationAutomated via Docker
Multi-clusterRequires multiple machines/VMsMultiple clusters on one machine

Think of it this way:

  • K3s = The Kubernetes distribution
  • K3d = The wrapper that runs K3s in Docker

How K3d Works​

Basic Workflow​

What Happens Behind the Scenes​

When you run k3d cluster create my-cluster:

  1. Docker Container Created: K3d creates a Docker container
  2. K3s Installed: K3s is started inside the container
  3. Networking Setup: Docker network created for cluster communication
  4. Kubeconfig Generated: K3d adds cluster config to ~/.kube/config
  5. LoadBalancer Created: Optional serverlb container for LoadBalancer services
  6. Ready: You can now use kubectl commands

K3d Components​

Components:

  • Server Container: Runs K3s control plane and workloads
  • Agent Container: (Optional) Runs workloads only
  • Serverlb Container: LoadBalancer proxy for accessing services
  • Docker Network: Connects all containers

K3d Features for Multi-Cluster​

1. Multiple Isolated Clusters​

Create as many clusters as you need:

k3d cluster create cloud
k3d cluster create fog  
k3d cluster create edge

Each cluster is completely isolated:

  • Separate API servers
  • Separate networks
  • Separate resources
  • Separate namespaces

2. Custom Networking​

Connect clusters on the same Docker network:

# Create shared network
docker network create multicluster

# Create clusters on shared network
k3d cluster create top --network multicluster
k3d cluster create mid --network multicluster
k3d cluster create bot --network multicluster

This allows:

  • Clusters can discover each other by container name
  • Direct communication between clusters
  • Simulates real network topology

3. Resource Simulation​

Simulate different resource tiers:

  • Cloud Tier
  • Fog Tier
  • Edge Tier
k3d cluster create cloud \
  --servers 1 \
  --agents 2

Simulates:

  • Large cluster
  • Multiple nodes
  • High resource availability
k3d cluster create fog \
  --servers 1 \
  --agents 1

Simulates:

  • Medium cluster
  • Limited nodes
  • Moderate resources
k3d cluster create edge \
  --servers 1 \
  --agents 0

Simulates:

  • Small cluster
  • Single node
  • Constrained resources

4. Port Mapping​

Expose services to your host machine:

k3d cluster create my-cluster \
  --api-port 6443 \
  -p "8080:80@loadbalancer" \
  -p "8443:443@loadbalancer"

Now services using ports 80/443 in the cluster are accessible on localhost:8080/8443.

5. Volume Mounts​

Share files between host and cluster:

k3d cluster create my-cluster \
  --volume /path/on/host:/path/in/cluster@server:0

Useful for:

  • Configuration files
  • Persistent data
  • Development code

Creating Your First K3d Cluster​

Installation​

  • Linux
  • macOS
  • Windows
curl -s https://raw.githubusercontent.com/k3d-io/k3d/main/install.sh | bash
brew install k3d
# Using Chocolatey
choco install k3d

# Or download from GitHub releases

Basic Commands​

# Create a cluster
k3d cluster create my-cluster

# List clusters
k3d cluster list

# Get kubeconfig
k3d kubeconfig get my-cluster

# Delete cluster
k3d cluster delete my-cluster

# Start stopped cluster
k3d cluster start my-cluster

# Stop cluster (preserves state)
k3d cluster stop my-cluster

Testing Your Cluster​

# Create cluster
k3d cluster create test

# Check nodes
kubectl get nodes

# Deploy test application
kubectl create deployment nginx --image=nginx

# Expose service
kubectl expose deployment nginx --port=80 --type=LoadBalancer

# Check service
kubectl get services

# Access nginx
curl localhost:8081  # or whatever port is mapped

Viewing Logs​

# K3s server logs
docker logs k3d-my-cluster-server-0

# Follow logs
docker logs -f k3d-my-cluster-server-0

# Check specific container
docker exec -it k3d-my-cluster-server-0 sh

Multi-Cluster Architecture with K3d​

Here's how we'll use K3d for cloud-fog-edge simulation:

Cluster Configuration:

ClusterRoleResources (Simulated)K3d Command
topCloud8 vCPU, 32GB RAM--servers 1
midFog4 vCPU, 16GB RAM--servers 1
botEdge2 vCPU, 4GB RAM--servers 1

Why This Works​

  1. Isolated but Connected: Each cluster is independent but can communicate via Docker network
  2. Realistic Simulation: Mimics real distributed architecture
  3. Fast Development: Create/destroy in seconds
  4. Cost Effective: No need for multiple VMs or cloud resources
  5. Reproducible: Same setup works on any machine with Docker

Common K3d Configurations​

Development Setup (Single Cluster)​

k3d cluster create dev \
  --servers 1 \
  --agents 2 \
  --api-port 6443 \
  -p "8080:80@loadbalancer" \
  -p "8443:443@loadbalancer" \
  --volume "${HOME}/projects:/projects@server:0"

What this does:

  • Creates cluster named "dev"
  • 1 server + 2 agent nodes
  • API accessible on port 6443
  • HTTP on localhost:8080, HTTPS on localhost:8443
  • Mounts your projects folder

Testing Setup (Multiple Clusters)​

# Create network
docker network create test-network

# Create clusters
for cluster in app-v1 app-v2 app-v3; do
  k3d cluster create $cluster \
    --servers 1 \
    --agents 0 \
    --network test-network \
    --api-port 0  # Random port
done

What this does:

  • Creates shared Docker network
  • Creates 3 independent clusters
  • All on same network (can communicate)
  • Random API ports (avoid conflicts)

CI/CD Setup​

k3d cluster create ci \
  --servers 1 \
  --agents 0 \
  --wait \
  --timeout 60s
  
# Run tests
kubectl apply -f tests/
kubectl wait --for=condition=complete job/test-job

# Cleanup
k3d cluster delete ci

What this does:

  • Quick cluster for CI pipeline
  • Waits for cluster to be ready
  • Runs tests
  • Cleans up after tests

Advanced K3d Features​

Registry Integration​

Run a local Docker registry with K3d:

# Create registry
k3d registry create my-registry --port 5000

# Create cluster using that registry
k3d cluster create my-cluster --registry-use k3d-my-registry:5000

Benefits:

  • Fast image pulls (local)
  • No Docker Hub rate limits
  • Test custom images

Image Import​

Import images directly into cluster:

# Build image
docker build -t my-app:latest .

# Import into cluster
k3d image import my-app:latest -c my-cluster

# Now deploy
kubectl create deployment my-app --image=my-app:latest

When to use:

  • Testing local images
  • No registry needed
  • Faster than pushing/pulling

Kubeconfig Management​

# Get kubeconfig for specific cluster
k3d kubeconfig get my-cluster > my-cluster-config.yaml

# Merge into default kubeconfig
k3d kubeconfig merge my-cluster --kubeconfig-merge-default

# Switch contexts
kubectl config use-context k3d-my-cluster

# View all contexts
kubectl config get-contexts

Cluster Snapshots​

Save and restore cluster state:

# Stop cluster (preserves state)
k3d cluster stop my-cluster

# Start again (restores state)
k3d cluster start my-cluster

# Delete and recreate (loses state)
k3d cluster delete my-cluster
k3d cluster create my-cluster

Troubleshooting​

Cluster Won't Start​

# Check Docker is running
docker ps

# Check for port conflicts
lsof -i :6443

# View K3d logs
k3d cluster list
docker logs k3d-my-cluster-server-0

# Delete and recreate
k3d cluster delete my-cluster
k3d cluster create my-cluster

Can't Access Services​

# Check LoadBalancer IP
kubectl get services

# Check port mappings
docker ps | grep k3d

# Test from inside cluster
kubectl run test --image=busybox -it --rm -- wget -O- http://my-service

Networking Issues​

# Check Docker network
docker network inspect k3d-my-cluster

# Check container connectivity
docker exec k3d-cluster1-server-0 ping k3d-cluster2-server-0

# Recreate network
docker network rm multicluster
docker network create multicluster

Out of Resources​

# Check Docker resource usage
docker stats

# Remove unused containers
docker container prune

# Remove unused images
docker image prune -a

# Increase Docker Desktop resources
# Docker Desktop → Settings → Resources

Best Practices​

1. Use Descriptive Cluster Names​

# Good
k3d cluster create prod-api
k3d cluster create staging-web
k3d cluster create test-db

# Bad
k3d cluster create cluster1
k3d cluster create test
k3d cluster create abc

2. Create Shared Networks for Related Clusters​

docker network create project-x

k3d cluster create frontend --network project-x
k3d cluster create backend --network project-x
k3d cluster create database --network project-x

3. Use Config Files for Complex Setups​

# k3d-config.yaml
apiVersion: k3d.io/v1alpha5
kind: Simple
metadata:
  name: my-cluster
servers: 1
agents: 2
ports:
  - port: 8080:80
    nodeFilters:
      - loadbalancer
options:
  k3s:
    extraArgs:
      - arg: --disable=traefik
        nodeFilters:
          - server:*
k3d cluster create --config k3d-config.yaml

4. Clean Up Regularly​

# Stop unused clusters
k3d cluster stop old-cluster

# Delete unused clusters
k3d cluster delete old-cluster

# Remove all clusters
k3d cluster delete --all

# Clean Docker
docker system prune

5. Document Your Clusters​

Keep a README with:

  • Cluster names and purposes
  • Port mappings
  • Special configurations
  • Dependencies between clusters

Performance Considerations​

Resource Usage​

Each K3d cluster uses:

  • CPU: ~0.5-1 vCPU idle, more under load
  • Memory: ~512MB minimum, 1-2GB comfortable
  • Disk: ~500MB per cluster

Recommendations:

  • Development machine: 2-3 clusters maximum
  • Powerful workstation: 5-10 clusters possible
  • CI/CD: Create/destroy as needed

Optimization Tips​

  1. Use agents sparingly: Single server node often sufficient
  2. Disable unused features: --k3s-arg "--disable=traefik@server:*"
  3. Limit running clusters: Stop clusters when not in use
  4. Use slim images: Keep container images small
  5. Monitor resources: docker stats to watch usage

When to Use K3s vs K3d​

Use K3s directly when:​

  • ✅ Deploying to production
  • ✅ Running on IoT devices or Raspberry Pi
  • ✅ Managing edge deployments
  • ✅ Need bare-metal performance
  • ✅ Long-running, permanent clusters

Use K3d when:​

  • ✅ Local development
  • ✅ Testing Kubernetes configurations
  • ✅ Learning Kubernetes
  • ✅ CI/CD pipelines
  • ✅ Multi-cluster development
  • ✅ Demo environments
